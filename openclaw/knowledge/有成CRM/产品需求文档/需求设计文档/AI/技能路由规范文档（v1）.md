## 第一部分　Skill 元数据模板

### 1.1 设计原则
一个 Skill 必须能被 **router（自动路由）** 和 **用户（手动触发）** 双重调用。因此它的元数据要同时服务两个读者：
- 给 **router 模型** 看 → 准确判断"该不该命中本技能"。
- 给 **用户** 看 → 看懂这技能能干啥（卡片描述、气泡文案）。

### 1.2 元数据字段（必备）

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| `skill_id` | string | 全局唯一，小写下划线 | `acquire.bid_search` |
| `name` | string | 显示名（用户语言） | 查近期招标 |
| `assistant` | string | 所属助理 | 获客拓客助理 |
| `scene` | enum | sales / scm | sales |
| `purpose` | string | **一句话职责（用户语言）**，≤20字 | 帮你找近期哪些单位在采购你们卖的东西 |
| `description` | string | **给 router 的技术描述**，含职责+边界+输入输出 | 按地区/产品/时间检索招投标平台并按匹配度排序。仅检索招标公告，不含企业工商查询 |
| `examples` | string[] | **few-shot 触发示例**，2–4 条用户原话 | ["查下杭州近期CRM招标","最近有政府机构采购客户管理系统吗"] |
| `boundary` | string[] | **反例**，明确"这不是我"+指明该去哪 | ["查某公司电话→走工商查询","查某公司历史招标→走企业背景"] |
| `parameters` | object[] | **参数槽位**（见 1.3） | 见下 |
| `side_effect_level` | enum | R / W / E | R |
| `confidence_policy` | enum | 引用第二部分策略 | `read_parallel` |
| `required_mcps` | string[] | 依赖的 MCP | ["bid_data","web_search"] |
| `output_format` | enum | markdown / json / card / record | markdown |
| `preconditions` | string[] | 权限/数据前置条件 | ["需开通招投标数据源"] |
| `on_failure` | enum | retry / fallback / notify | notify |
| `reversible` | bool | 是否可撤销（影响确认强度） | true |

### 1.3 参数槽位（parameters）子结构
每个参数含：`name` / `type`(string|date|enum|number) / `required` / `default` / `enum` / `description` / `ask_if_missing`(缺省时是否反问)。

```yaml
parameters:
  - { name: region, type: string, required: true,  ask_if_missing: true, description: "地区，如 杭州/浙江省" }
  - { name: product, type: string, required: true, ask_if_missing: true, description: "产品关键词" }
  - { name: days,    type: number, required: false, default: 30,         description: "时间范围（天）" }
  - { name: org_type,type: enum,   required: false, default: all, enum: [all,gov,enterprise], description: "项目类型" }
```

### 1.4 完整模板（可直接复制填充）
```yaml
skill:
  skill_id:
  name:
  assistant:
  scene:           # sales | scm
  purpose:         # 一句话职责，用户语言，≤20字
  description:     # 给router：职责 + 边界 + 输入输出
  side_effect_level:   # R | W | E
  confidence_policy:   # 见第二部分
  reversible:          # true | false
  output_format:       # markdown | json | card | record

  examples:            # few-shot 正例
    - 
  boundary:            # 反例 + 指明正确去处
    - 

  parameters:
    - { name: , type: , required: , default: , ask_if_missing: , description: }

  required_mcps:
    - 
  preconditions:
    - 
  on_failure:      # retry | fallback | notify
```

### 1.5 写作规范（硬性要求）
- **purpose 用"帮你…"句式**，禁止术语（"检索招投标数据库"❌ → "帮你找近期哪些单位在采购你们卖的东西"✅）。
- **examples 必须是用户真实原话**，不是规格说明（"查询招投标信息"❌ → "查下杭州近期CRM招标"✅）。
- **boundary 至少 2 条**，且每条必须指向"该去哪"——反例没有去处的，等于没写。
- **description 要包含否定边界**（"仅…不含…"），这是降低误路由最有效的写法。
- **side_effect_level 必须填**，它直接决定第三部分的确认强度。

## 第二部分　置信度分档、阈值与各档交互

### 2.1 分档定义（router 对每个候选技能打分 0–1）

| 档位 | 阈值 | 含义 |
|---|---|---|
| **高置信** | ≥ 0.80 | 意图明确，技能命中无疑 |
| **中置信** | 0.50 – 0.80 | 大概率对，但存在歧义或参数缺失 |
| **低置信** | < 0.50 | 不确定，可能是别的技能或不属于本助理 |

> 阈值可由企业配置：严控企业上调（高≥0.85），宽松企业下调（高≥0.70）。

### 2.2 处理策略矩阵（置信度 × 副作用等级）

这是核心规则——**确认强度同时取决于"我有多确定"和"错了多严重"**：

| | R 只读/查询 | W 写库/状态变更 | E 对外/不可逆 |
|---|---|---|---|
| **高置信** | ✅ **直接执行**，免确认 | ⚠️ **轻确认**（一句话复述意图后执行） | 🔴 **强确认**（明确列出将执行动作+留痕） |
| **中置信** | 🔵 **多路并行召回**相关技能，结果合并排序后返回 | ⚠️ **复述 + 参数反问补全**，确认后执行 | 🔴 **强确认 + 参数反问** |
| **低置信** | 🔵 多路召回；若全部极低→**列出候选让用户选** | 🔴 **不自动执行**，列出候选/反问 | 🔴 **绝不自动执行**，列出候选让用户明确选择 |

**三条铁律：**
1. **只读(R)类永不需要"是否执行"的确认**——它幂等无副作用，错了重说即可。最多做"多路召回"对冲不确定性。
2. **对外(E)类任何置信度都要确认**——哪怕高置信，因为它不可逆、有外部影响。
3. **置信度越低、副作用越大 → 选择权越要交还用户**（列候选/反问），而不是"硬猜一个结果给用户"。

### 2.3 提升置信度的手段（减少打断）
不是所有"中/低置信"都要靠问用户，能用以下手段把置信度顶上去的，先顶：
- **参数补全**：从上下文/默认值补全缺失参数（如"近期"→默认30天），补全后置信度↑。
- **历史偏好**：用户上次选过"政府+企业都看"，本次默认沿用，置信度↑。
- **同义归一**："中标信息/招投标/采购公告"归一到同一技能，召回↑。

只有以上都用尽仍达不到高置信，才进入"反问/列候选"。

### 2.4 多路并行召回的规则（R 类中低置信专用）
当只读类意图不明确时，**不要猜一个，把语义相关的几个技能都跑掉**，再合并：
- 并行执行置信度 > 0.3 的候选技能；
- 结果按各技能置信度 × 结果相关性排序；
- 每条结果标注来源技能，让用户一眼分辨；
- 用户从结果里点哪个，等于"免费"喂给 router 一个正确范例（下次置信度↑）。

## 第三部分　确认话术规范

### 3.1 总原则
1. **只在 W / E 类、且置信度不足"直接执行"时确认**。只读类不确认（确认只读是过度设计）。
2. **说"将要做什么"，不说"我理解了什么"**。用户关心结果，不关心你的理解过程。
3. **一个动作一句话**，关键参数加粗，**一键确认/取消**，不要成段。
4. **批量动作可勾选**：发现 5 条待办，默认全选，允许用户取消勾选个别项，而不是逐条确认。
5. **可逆性要说清**：不可逆动作必须标注（"此操作不可撤销"）。
6. **不重复骚扰**：同一会话内用户已确认过的同类重复动作，给出"对后续同类都这样处理"的选项（学习偏好）。

### 3.2 各场景话术模板

#### A. 轻确认（W 类 · 高置信）
> 格式：`我将 [动作]，[关键参数]。确认吗？`
- ✅ 我将把"发送CRM报价单"记为一条跟进（计划 6/22，跟进人：我）。确认创建吗？
- ✅ 我将生成一份「杭州·CRM·近30天」的报价单草稿。确认生成吗？

#### B. 复述 + 参数反问（W 类 · 中置信，参数缺失）
> 格式：`我准备 [动作]。另外：[缺失参数提问]。`
- ✅ 我准备按"杭州·CRM·近30天·政府+企业"检索招标。时间范围按近 30 天可以吗？需要排除已过截止日的吗？

#### C. 强确认（E 类 · 任意置信，对外/不可逆）
> 格式：`即将 [对外动作]，[金额/数量/对象]。[不可逆提示]。确认提交？`
- ✅ 即将向「华东建材」发起采购订单：¥120,000，3 条明细。**此操作将通知供应商、不可撤销**。确认提交吗？
- ✅ 即将对 8 名审批人发起合同审批通知。确认发起？

#### D. 列候选（低置信，不确定是哪个技能）
> 格式：`你是想：①… ②… ③… ？直接回复序号或补充说明。`
- ✅ 你是想：① 查近期招标 ② 查某家公司背景 ③ 把线索入库？直接回复序号或补充说明。

#### E. 批量确认（抽取类技能产出多条）
> 格式：`发现 N 条 [类型]，默认全选。可取消勾选。[一键创建]`
- ✅ 从这段沟通发现：3 条待跟进、1 个问题。默认全部创建。你可以取消勾选不需要的。
  - ☑ 发送CRM报价单（6/22）
  - ☑ 确认预算到位时间（6/20）
  - ☑ 移动端闪退 → 工单 P1
  - [ 创建全部 ]

### 3.3 反例（禁止写法）
| ❌ 错误写法 | 问题 | ✅ 正确 |
|---|---|---|
| "我理解您要查询招投标信息，是否确认执行？" | 确认只读操作（R类不该确认） | 直接执行，不确认 |
| "系统将调用 bid_search 技能" | 露技术术语/技能ID | 隐藏技能名，说结果 |
| "请确认您的操作" | 没说要做什么 | "我将把…记为跟进，确认吗？" |
| "您确定吗？确定请点击确定" | 无信息量、无参数 | 列出动作+参数 |
| 5条待办逐条弹窗确认 | 打断过重 | 批量勾选一次确认 |

### 3.4 文案语气
- 用"我将…/即将…"，不用"系统将…"（拟人，去机器感）。
- 数字、日期、对象**加粗或单列**，便于扫读。
- 危险动作（删除、对外、付款）用**警示色**与普通动作区分。
- 取消按钮始终可达，且取消后说明"已取消，未做任何改动"。

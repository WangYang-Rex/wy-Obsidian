## 1\. 需求背景

### 1.1 需求描述

有成报销单据导出支付格式时，中国银行转账（对私/通用）模板（ABCTransfer）缺少**收款方开户行**、**付款账号**和**付款户名**三个关键字段，导致出纳导出后需手工补录，效率低且易出错。本次需求将这三个字段补充至对私/通用模板，使导出文件可直接用于银行批量转账，无需二次编辑。

### 1.2 需求背景
- **需求方**：出纳（财务人员）
- **现状问题**：
• 中国银行有3个导出模板，「对公支付」模板（ABCForPublicTransfer）已包含付款账号、收款方开户行和付款户名
• 但对私/通用模板（ABCTransfer）仅8个字段，缺少这三个关键字段
• 出纳导出后需手工补录，效率低易出错（预计每次导出需额外5-10分钟）
• 其他银行模板（农业银行等）不受影响
- **解决思路**：在ABCTransfer模板中新增这三个字段，数据来源与对公模板保持一致，字段顺序优化为"付款信息在前，收款信息在后"

**业务流程**
```
单据审批完成 → 出纳导出支付格式 → 选择中国银行对私/通用模板 → 下载导出文件 → 网银批量导入 → 完成付款

```

## 2\. 功能现状

### 2.1 功能位置

单据列表 → 导出 → 支付格式 → 中国银行 → 对私/通用模板（ABCTransfer）

支持导出的页面（通过 `showOnlineBankingInExportModal` 配置控制）：
- 审批 \> 待审批（pendingA）
- 审批 \> 待支付（unpaid）
- 单据管理 \> 报销管理（reimburse）
- 单据管理 \> 借款管理（loanMana）
- 单据管理 \> 付款管理（paymentM）

### 2.2 支持的单据类型

仅报销单(type=1)、借款单(type=2)、付款单(type=3)支持支付格式导出。

### 2.3 中国银行模板对比

| 模板名称 | 类名 | 字段数量 | 付款账号 | 付款户名 | 收款方开户行 |
|------------|------|------------|------------|------------|------------------|
| 中国银行转账（对私/通用） | ABCTransfer | 8个 | ❌ 无 | ❌ 无 | ❌ 无 |
| 中国银行对公支付 | ABCForPublicTransfer | 13个 | ✅ 有 | ✅ 有 | ✅ 有 |
| 中国农业银行转账 | RealABCTransfer | 8个 | - | - | - |

### 2.4 当前导出逻辑

**核心逻辑**：一行付款信息 = 导出表格中的一行

**合并收款人逻辑**（megreCollectionFlag = true）：
- 合并条件：相同银行卡号 \+ 相同银行名（bankCardNum \+ bankName）
- 金额：累加计算
- 附言：逗号拼接多个单据名称
- 其他字段：取第一次出现的值

**当前字段（8个）**：账号、户名、金额、账号所在省、证件类型、证件号码、用途、附言

### 2.5 缺失能力
- 缺少收款方开户行（收款人开户行）
- 缺少付款账号（付款人账号）
- 缺少付款户名

### 3.2 详细功能描述

#### 3.2.1 字段扩展与顺序优化

**位置**：单据列表 → 导出 → 支付格式 → 中国银行 → 对私/通用模板

**调整后字段（共11个）**

| 序号 | 导出字段 | 变更 | 字段含义 | 取值来源 |
|------|------------|------|------------|------------|
| 1 | 付款账号 | 新增 | 付款账户银行账号 | 出纳节点选择付款账号的银行账号 |
| 2 | 付款户名 | 新增 | 付款账户户名 | 出纳节点选择付款账号的户名 |
| 3 | 收款账号 | - | 收款人银行账号 | 单据收款信息-银行账号 |
| 4 | 收款户名 | - | 收款人姓名 | 单据收款信息-户名 |
| 5 | 收款方开户行 | 新增 | 收款人开户银行全称 | 单据/员工档案/供应商档案 |
| 6 | 金额 | - | 转账金额 | 单据支付金额（元） |
| 7 | 账号所在省 | - | 收款人所在省份 | 银行账号解析或档案配置 |
| 8 | 证件类型 | - | 证件类型 | 员工档案/供应商档案 |
| 9 | 证件号码 | - | 证件号码 | 员工档案/供应商档案 |
| 10 | 用途 | - | 交易用途 | 基础配置或自定义摘要 |
| 11 | 附言 | - | 交易附言 | 单据名称（多单据逗号拼接） |

**字段分组**
- 第1-2列：付款信息（新增付款账号、付款户名）
- 第3-9列：收款信息（含新增收款方开户行）
- 第10-11列：交易信息

#### 3.2.2 取值规则

**付款账号取值规则**

| 场景 | 取值来源 | 为空处理 |
|------|------------|------------|
| 单据已到出纳节点且选择了付款账号 | 出纳节点选择的付款账号（银行账号） | - |
| 单据未到出纳节点 | - | 导出空字符串 |
| 出纳未选择付款账号 | - | 导出空字符串 |
| 多账户付款 | 分别取每一个付款账号，每个付款账号独立生成一行 | - |

**付款户名取值规则**

| 场景 | 取值来源 | 为空处理 |
|------|------------|------------|
| 单据已到出纳节点且选择了付款账号 | 出纳节点选择的付款账号对应户名 | - |
| 单据未到出纳节点 | - | 导出空字符串 |
| 出纳未选择付款账号 | - | 导出空字符串 |
| 多账户付款 | 分别取每一个付款账号对应的户名 | 与付款账号保持一致，每行对应一个付款账号 |

**收款方开户行取值规则**

| 优先级 | 数据来源 | 取值字段 | 为空处理 |
|---------|------------|------------|------------|
| 1 | 单据收款信息 | 开户行（手动填写） | - |
| 2 | 报销人基础档案 | 银行开户行 | - |
| 3 | 供应商基础档案 | 银行开户行 | - |
| 4 | 默认值 | - | 导出空字符串 |

**数据格式处理** • 直接取档案/单据中存储的原始值，不做截断或格式转换
• 超长内容（超过100字符）仍完整导出，由银行端处理

#### 3.2.3 合并收款人逻辑

**核心原则**：合并仅针对满足条件的记录，付款账号不一致时不合并，避免付款账户混乱。

**合并触发条件** • 银行卡号相同（bankCardNum）
• 开户银行名称相同（bankName，即收款方开户行）
• **付款账号一致**（新增约束，不一致时不合并）

**合并处理规则**

| 字段 | 合并处理逻辑 |
|------|------------------|
| 金额 | 累加计算（多行金额相加求和） |
| 附言 | 逗号拼接多个单据名称 |
| 收款方开户行 | 取合并组第一行（理论上各行的收款方开户行相同） |
| 付款账号 | 取合并组第一行 |
| 付款户名 | 取合并组第一行 |
| 其他字段 | 取合并组第一行 |

**合并场景示例**

**场景1：正常合并（条件都满足）**
```
单据A-行1：张三，工行，1000元，付款账号：9558880001
单据A-行2：张三，工行，1000元，付款账号：9558880001（与行1合并）

合并后：张三，工行，2000元，付款账号：9558880001，附言：单据A,单据A

```

**场景2：付款账号不一致，不合并**
```
单据A：张三，工行，1000元，付款账号：9558880001
单据B：张三，工行，2000元，付款账号：9558880002（不同付款账号）

处理结果：不合并，保持两行独立导出
行1：张三，工行，1000元，付款账号：9558880001，附言：单据A
行2：张三，工行，2000元，付款账号：9558880002，附言：单据B

```

### 3.3 逻辑互斥与边界处理

**边界情况处理**

| 边界场景 | 触发条件 | 处理方案 |
|------------|------------|------------|
| 数据为空-收款方开户行 | 档案未维护开户行 | 导出空字符串，不阻断导出，出纳可手工补录 |
| 数据为空-付款账号 | 未到出纳节点/未选付款账号 | 导出空字符串，不阻断导出 |
| 数据为空-付款户名 | 未到出纳节点/未选付款账号 | 导出空字符串，不阻断导出 |
| 合并-付款账号不同 | 合并组内付款账号不一致 | **不合并**，保持独立行导出，避免付款账户混乱 |
| 合并-收款方开户行不同 | bankName不同（如不同支行） | **不合并**，保持独立行导出 |
| 多账户付款 | 一笔单据多个付款账号 | 分别导出为多个独立行，每行对应一个付款账号及其户名 |
| 超长内容 | 开户行名称超过100字符 | 完整导出，不做截断，由银行端处理 |
| 存量兼容 | 历史已导出记录 | 不受本次变更影响，再次导出时按新逻辑执行 |

**与其他功能的兼容性**

| 相关功能 | 交互关系 | 处理方案 |
|------------|------------|------------|
| 对公模板(ABCForPublicTransfer) | 独立模板 | 保持原逻辑不变，不受影响 |
| 农业银行模板(RealABCTransfer) | 独立模板 | 保持原逻辑不变，不受影响 |
| 自定义摘要和用途 | 配置项 | 出纳使用了时优先取自定义值，未自定义时取默认值 |

## 附录

**A. 典型使用场景**

**场景**：出纳批量支付员工差旅费报销

**配置步骤（首次）**：
1. 进入基础档案 → 收款信息，确认公司付款账号信息完整（账号、户名）
2. 进入员工档案，确认员工银行信息完整（银行账号、开户行）

**使用步骤（日常）**：
1. 进入单据管理 → 报销管理，筛选待支付的差旅费报销单
2. 勾选需要支付的单据，点击导出 → 支付格式导出
3. 选择中国银行 → 对私/通用模板
4. 下载导出文件，确认字段完整（付款账号、付款户名、收款方开户行均已填充）
5. 登录中国银行网银，批量导入导出文件
6. 核对导入信息，确认无误后提交付款

**预期效果**：导出文件直接可用，无需手工补录付款账号和开户行信息，单次导出节省5-10分钟。

## 1\. 需求背景

### 1.1 需求描述

有成报销已支持绑定邮箱自动拉取邮件中的发票并生成消费记录，目前支持的邮件类型包括：
- 附件型发票邮件（通用，如滴滴）
- 正文链接型发票邮件（诺诺网、阿里发票平台）

曹操出行（曹操专车）作为主流出行平台之一，其电子发票邮件采用**正文HTML下载链接**的形式发送，且同时提供PDF、OFD、XML三种发票格式及独立行程单链接。当前系统无法识别和处理曹操出行的发票邮件，导致使用曹操专车的企业员工需要手动下载发票并上传至报销系统。

本次需求在现有绑定邮箱获取发票生成消费的功能基础上，新增对曹操专车发票邮件的兼容支持，使曹操专车发票的采集流程与滴滴、高德等现有平台保持一致。

### 1.2 需求背景

**业务痛点**：
1. **曹操专车发票无法自动采集**：曹操出行发票邮件为正文链接型，不在现有支持的诺诺网/阿里发票平台之列，系统自动忽略，员工需手动处理。
2. **多格式邮件增加复杂度**：曹操邮件同时提供PDF/OFD/XML三种格式及行程单，员工不确定该下载哪种格式，容易重复下载或遗漏。
3. **行程单缺失影响报销完整性**：财务报销通常要求发票\+行程单作为完整凭证，员工容易遗漏行程单下载。

**用户场景**：
- **企业员工**：使用曹操专车后，绑定企业邮箱，系统自动拉取发票和行程单，无需手动操作。
- **财务管理员**：曹操专车发票与其他平台一样自动进入报销系统，员工不再遗漏。

### 1.3 业务流程
```
【曹操专车发票自动采集流程】

1. 员工使用曹操专车并完成订单
   ↓
2. 曹操出行发送电子发票邮件到员工绑定邮箱
   （发件人：notify@mail.caocaokeji.cn，主题：曹操出行电子发票）
   ↓
3. 有成报销 EmailPullTask（每天凌晨1点）拉取邮件
   ↓
4. 识别曹操专车邮件（发件人含"caocao" + 标题含"发票"）
   ↓
5. 解析HTML正文，提取下载链接：
   ├─ 发票PDF链接 → 下载 → 上传OSS → email_attachment（invoice_status=0）→ 走OCR
   ├─ 发票OFD链接 → 如PDF已下载则跳过
   ├─ 发票XML链接 → 如PDF/OFD已下载则跳过
   └─ 行程单链接 → 下载 → 上传OSS → 文件名加"曹操出行行程单_"前缀 → 跳过OCR
   ↓
6. OCR识别（MailAttachmentToInvoiceTask）
   ├─ 发票PDF → financeInvoiceList.rjson → invoice_status=1
   └─ 行程单 → shouldIgnoreAttachmentForMerge() 跳过OCR
   ↓
7. 发票转消费（MailInvoiceToFeeTask）
   ├─ 单条模式：每张发票生成一条消费，行程单挂载为附件
   └─ 合并模式：同一 emailPullId 下发票+行程单合并为一条消费
   ↓
8. 员工在报销系统中查看已生成的消费记录（含发票+行程单）

```

## 2\. 功能现状

### 2.1 邮箱拉取发票现有流程

有成报销已支持绑定邮箱自动拉取发票生成消费，三阶段异步流水线：

| 阶段 | 定时任务 | 处理类 | 核心职责 |
|------|------------|---------|------------|
| 阶段一（邮件拉取） | EmailPullTask | EmailTaskProcessServiceImpl.doEmailPullAllTask() | IMAP连接邮箱，拉取最近2天内标题含"发票"/"电子票"的邮件 |
| 阶段二（OCR识别） | MailAttachmentToInvoiceTask | EmailAttachmentServiceImpl.handleeAttachmentTask(INIT) | 下载附件，调用 financeInvoiceList.rjson 识别发票 |
| 阶段三（消费生成） | MailInvoiceToFeeTask | EmailAttachmentServiceImpl.handleInvoiceToFeeTask(WAIT) | 按用户配置的单条/合并模式生成消费记录 |

**现有邮件处理类型**：

| 邮件类型 | 现有支持平台 | 处理逻辑 |
|------------|------------------|------------|
| 附件型 | 通用（滴滴等） | 提取 Multipart BodyPart → 按 PDF \> OFD \> XML 优先级仅保存一种 → 上传OSS |
| 正文链接型 | 诺诺网、阿里发票平台 | 解析HTML获取下载链接 → 按 PDF \> OFD \> XML 优先级仅保存一种 → 下载 → 上传OSS |

### 2.2 正文链接型邮件现有支持

| 平台 | 识别方式 | 处理逻辑 |
|------|------------|------------|
| 诺诺网 | 发件人含"诺诺网" | 解析HTML获取发票下载链接 |
| 阿里发票平台 | 发件人含"阿里发票平台" | 解析HTML获取发票下载链接 |
| 其他平台 | — | ❌ 不处理，发票无法提取 |

### 2.3 多格式优先级处理（现有逻辑）

当同一封邮件中，同一个发票存在多种格式（PDF \+ OFD \+ XML）时，按以下优先级仅保存一种：
```
PDF > OFD > XML

```

| 优先级 | 格式 | 说明 |
|---------|------|------|
| 1 | PDF | 首选，优先下载并保存 |
| 2 | OFD | 无PDF时，下载OFD |
| 3 | XML | 无PDF且无OFD时，下载XML |
> 判定"同一发票"：同一封邮件（emailPullId）下，附件文件名（去除扩展名后）相同或高度相似。   

### 2.4 行程单现有处理逻辑

合并模式（`emailInvoiceMergeToFee == 1`）下：
- 附件文件名含"行程" 或 "高德打车\+行程单" → `shouldIgnoreAttachmentForMerge()`（约第200行）跳过OCR
- 被跳过的附件 `invoiceId = null`，在 `invoiceToFeeMerge()`（约第290行）中被收集到 `attachmentList`
- 最终作为非发票附件挂载到合并消费记录的附件组件（`ComponentTypeEnum.ATTACHMENT`）

### 2.5 关键数据表

| 表名 | 分表数 | 核心字段 | 说明 |
|------|---------|------------|------|
| `email&#95;bind&#95;record` | 8 | 用户ID、邮箱地址、IMAP配置 | 用户邮箱绑定配置 |
| `email&#95;pull&#95;record` | 无 | `from&#95;user&#95;name`、邮件标题 | 邮件拉取记录，可追溯发件人 |
| `email&#95;attachment` | 32 | `invoice&#95;status`、`fee&#95;status`、`invoice&#95;id`、`attachment` | 附件记录，含OCR和消费生成状态 |

### 3.2 详细功能描述

#### 3.2.1 曹操专车邮件识别

**位置**：阶段一（邮件拉取）— `EmailTaskProcessServiceImpl.doEmailPullAllTask()`（邮件任务处理服务-执行全部邮件拉取任务）

**新增识别规则**：

| 判断维度 | 规则 | 优先级 |
|------------|------|---------|
| 发件人 | 包含 **"caocao"**（不区分大小写）或 **"曹操出行"** | 必须满足 |
| 邮件标题 | 包含 **"发票"** 或 **"电子票"**（复用现有标题过滤） | 必须满足 |
> 双重校验：发件人含"caocao"/"曹操出行" **且** 标题含"发票"/"电子票"，才判定为曹操出行发票邮件。缺一不可，降低误判率。   

#### 3.2.2 HTML正文解析与多格式优先级处理

**解析目标**：提取邮件正文HTML中的下载链接，分两类处理：

**第一类：发票文件链接**

| 链接类型 | 提取关键词 | 处理方式 |
|------------|---------------|------------|
| 发票PDF文件 | 正文含"发票PDF文件"或类似锚点 | 下载 → 上传OSS → `email&#95;attachment`（`invoice&#95;status=0`）→ 走OCR |
| 发票OFD文件 | 正文含"发票OFD文件"或类似锚点 | **按优先级判断**：如PDF已下载则跳过，否则下载 |
| 发票XML文件 | 正文含"发票XML文件"或类似锚点 | **按优先级判断**：如PDF/OFD已下载则跳过，否则下载 |

**多格式优先级处理规则**：
```
同一发票多格式时：PDF > OFD > XML，仅保存优先级最高的一种

示例：
- 同时存在 PDF + OFD + XML → 仅下载 PDF
- 仅存在 OFD + XML（无PDF）→ 仅下载 OFD
- 仅存在 XML（无PDF/OFD）→ 下载 XML

```

**判定"同一发票"的方式**：
- 同一封邮件（`emailPullId`）下
- 链接文本或URL中包含相同的发票标识（如发票号码、订单号等）
- 或链接锚点文本相似度高（如均含"发票"关键字，排除"行程单"）

**第二类：行程单链接**

| 链接类型 | 提取关键词 | 处理方式 |
|------------|---------------|------------|
| 所关联行程单 | 正文含"所关联行程单"或类似锚点 | 下载 → 上传OSS → 文件名处理 → `email&#95;attachment` → **跳过OCR** |

#### 3.2.3 行程单文件名处理

**目的**：确保行程单能被现有 `shouldIgnoreAttachmentForMerge()`（合并模式附件忽略判断，约第200行）规则识别，复用现有逻辑，无需新增OCR跳过代码。

**处理规则**：
- 下载行程单文件后，保存到 `email_attachment` 时检查原始文件名
- 若原始文件名**不含"行程"或"行程单"**：在文件名前追加前缀 **"曹操出行行程单\_"**
- 若原始文件名**已含"行程"或"行程单"**：保持原文件名不变

**示例**：

| 原始文件名 | 处理后文件名 | 说明 |
|---------------|------------------|------|
| `2740482198513913.pdf` | `曹操出行行程单&#95;2740482198513913.pdf` | 追加前缀，确保被识别 |
| `trip&#95;20260324.pdf` | `曹操出行行程单&#95;trip&#95;20260324.pdf` | 追加前缀 |
| `曹操出行行程单.pdf` | `曹操出行行程单.pdf` | 已含关键字，不变 |

#### 3.2.4 行程单OCR跳过

**位置**：阶段二（OCR识别）— `EmailAttachmentServiceImpl.handleeAttachmentTask()`（邮件附件服务-处理附件任务）

**触发条件**：
- 文件名含"曹操出行行程单"或"行程单"
- 且关联的 `email_pull_record.from_user_name`（邮件拉取记录-发件人名称）含"caocao"

**处理逻辑**：
- **直接跳过OCR**，不调用 `financeInvoiceList.rjson`（OCR发票识别接口）
- `invoice_status`（发票状态）保持初始值（不更新）
- `fee_status`（消费状态）设为 **WAIT(3)**
- 进入阶段三消费生成时，作为 `attachmentList`（附件列表）被挂载到消费记录
> 注：由于文件名经3.2.3处理后已含"行程单"，可被现有 `shouldIgnoreAttachmentForMerge()` 自动识别并跳过OCR（合并模式下）。单条模式下同理。   

#### 3.2.5 消费生成（复用现有逻辑）

**单条模式**（`emailInvoiceMergeToFee`（邮件发票合并转消费配置）== 0）：
- 发票PDF → 正常OCR → `SingleCreateConsumptionInfoRequest` → 生成一条消费记录
- 行程单 → 跳过OCR → 作为附件挂载到该条消费记录

**合并模式**（`emailInvoiceMergeToFee`（邮件发票合并转消费配置）== 1）：
- 以 `emailPullId`（邮件拉取ID）为维度，同一封曹操邮件的发票PDF \+ 行程单合并为**一条消费记录**
- 发票PDF → OCR成功 → `MergeCreateConsumptionInfoRequest` → 发票组件数据
- 行程单 → `invoiceId`（发票ID）= null → 收集到 `attachmentList`（附件列表）→ 附件组件数据

### 3.3 边界处理

| 序号 | 边界场景 | 处理规则 |
|------|------------|------------|
| 1 | 曹操邮件同时提供PDF\+OFD\+XML | 按PDF\>OFD\>XML优先级，仅下载PDF，其余跳过 |
| 2 | 曹操邮件仅提供OFD（无PDF） | 下载OFD，正常走OCR流程 |
| 3 | 曹操邮件仅提供XML（无PDF/OFD） | 下载XML，正常走OCR流程 |
| 4 | 曹操邮件无任何发票链接 | 不下载发票文件，记录日志，标记为无发票 |
| 5 | 曹操邮件无行程单链接 | 仅下载发票，正常生成消费，无行程单附件 |
| 6 | 曹操行程单链接失效（\>72小时） | 行程单下载失败，记录日志，不影响发票PDF处理 |
| 7 | 曹操邮件发件人大小写混合 | 正常识别（不区分大小写匹配"caocao"） |
| 8 | 同一邮箱同时收到曹操\+滴滴邮件 | 各自独立处理，互不干扰 |
| 9 | OCR接口识别曹操发票失败 | `invoice&#95;status`（发票状态）= 2，进入失败重试队列 |
| 10 | 存量已绑定邮箱（非曹操） | 不受影响，正常拉取 |
| 11 | 诺诺网/阿里平台邮件 | 不受影响，正文链接解析逻辑正常 |
| 12 | 高德多格式附件邮件 | 不受影响，现有PDF\>OFD\>XML优先级逻辑正常 |

### D. 典型使用场景

**场景一：曹操专车标准发票采集**
> 员工使用曹操专车完成一次出行，收到曹操出行电子发票邮件（含PDF\+OFD\+XML\+行程单）。   

流程：
1. 凌晨1点，EmailPullTask 拉取到该邮件
2. 识别发件人含"caocao" \+ 标题含"发票"，判定为曹操专车邮件
3. 解析HTML，提取4个链接：PDF发票、OFD发票、XML发票、行程单
4. 按PDF\>OFD\>XML优先级，仅下载PDF发票，OFD和XML跳过
5. 下载行程单，文件名追加前缀"曹操出行行程单\_"
6. 发票PDF走OCR识别，行程单被跳过OCR
7. 合并模式下，同一 emailPullId 的发票\+行程单合并为一条消费记录
8. 员工在报销系统中看到消费记录，含发票组件和行程单附件

### E. 改动范围总结

| 阶段 | 改动点 | 工作量评估 |
|------|---------|---------------|
| 阶段一（邮件拉取） | 新增曹操发件人识别 \+ HTML解析（提取PDF/OFD/XML链接\+行程单链接）\+ 多格式优先级处理 \+ 行程单文件名处理 | 中等 |

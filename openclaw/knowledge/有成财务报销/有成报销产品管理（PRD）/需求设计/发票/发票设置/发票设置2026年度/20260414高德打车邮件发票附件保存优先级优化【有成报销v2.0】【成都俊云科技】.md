## **1\. 需求背景**

### **1.1 需求描述**

为邮件拉取发票的附件保存逻辑增加优先级控制能力，在"同一封邮件内的发票和行程单合并为一笔消费"开启时，同一发票的多格式附件（pdf、ofd、xml）按 pdf \> ofd \> xml 优先级保存一份到消费中，提升用户查看发票附件的体验。

### **1.2 需求背景**
- 需求方：使用邮件拉取发票的企业员工及财务管理员
- 现状问题：
    - 系统从邮件中获取发票附件时，同一张发票可能包含pdf、ofd、xml三种格式文件
    - 开启"同一封邮件内的发票和行程单合并为一笔消费"后，这三个格式文件的发票仅会合并到一个消费中，且仅保存一个格式的发票文件
    - 当前保存的是xml格式附件，不便于用户直接查看发票票面信息
- 解决思路：
    - 在附件保存阶段引入优先级判断规则：pdf \> ofd \> xml
    - 合并消费模式下仅保存优先级最高的一份发票附件

业务流程
```
系统从邮件拉取发票附件
    ↓
识别同一发票关联的多个格式文件（pdf、ofd、xml）
    ↓
判断"发票和行程单合并为一笔消费"开关状态
    ↓
若开启：按 pdf > ofd > xml 优先级保存一份附件到消费中
若关闭：各格式附件分别生成独立消费并各自存储
    ↓
员工在消费详情页查看已保存的附件
```

## **2\. 功能现状**

功能位置：邮箱绑定-发票自动生成消费（全局设置-费用设置-同一封邮件内的发票和行程单合并为一笔消费）

现有能力：
- 系统支持从邮件中自动拉取发票附件
- 支持"同一封邮件内的发票和行程单合并为一笔消费"开关配置
- 开启合并后，同一发票的三个格式文件（pdf、ofd、xml）仅会合并到一个消费中
- 当前合并后仅保存xml格式附件到消费中

缺失能力：
- 不支持按 pdf \> ofd \> xml 优先级选择保存的发票附件格式

### **3.2 详细功能描述**

#### 3.2.1 邮件发票附件保存优先级优化

位置：邮件拉取发票服务 → 附件解析与保存逻辑

新增配置/规则
- 系统从邮件中解析同一发票的多个格式附件（pdf、ofd、xml）时，新增优先级判断规则
- 优先级顺序固定为：pdf \> ofd \> xml

交互规则
- 仅在"同一封邮件内的发票和行程单合并为一笔消费"开关开启时生效
- 开关关闭时，保持现有逻辑不变
- 若同一发票仅存在一种格式附件，则直接保存该格式

核心逻辑说明
- 系统从邮件中拉取发票附件后，识别同一发票关联的多个格式文件
- 判断合并开关状态：
    - 开启：按优先级选取最高的一份附件关联到生成的消费中
    - 关闭：各格式附件分别生成独立消费并各自存储

业务流程
```
系统从邮件拉取发票附件
    ↓
识别同一发票的多个格式文件
    ↓
判断合并开关状态
    ├─ 开启 → 按 pdf > ofd > xml 保存一份 → 生成消费
    └─ 关闭 → 各格式分别生成独立消费
    ↓
消费详情页展示已保存附件
```

#### 3.2.2 消费详情附件展示

位置：我的-待办-消费详情页

核心原则：消费详情页直接展示该消费下已保存的发票附件，展示层不做额外的优先级筛选。

展示规则
- 新拉取生成的消费：详情页展示按优先级保存的那份附件
- 存量消费：详情页仍展示已保存的附件（如存量为xml则继续展示xml）
- 若消费下无附件，展示区域保持为空

### **3.3 逻辑互斥与边界处理**

边界情况处理

| 边界场景 | 触发条件 | 处理方案 |
|-------------------------------------------------------------------------------|-------------------------------------------------------------------------------|-------------------------------------------------------------------------------|
| 单一格式 | 同一发票仅存在xml格式 | 保存xml格式 |
| 两格式共存 | 同一发票有pdf和xml，无ofd | 保存pdf格式 |
| 两格式共存 | 同一发票有ofd和xml，无pdf | 保存ofd格式 |
| 三格式共存 | 同一发票同时有pdf、ofd、xml | 保存pdf格式 |
| 存量兼容 | 历史已生成的邮件消费 | 保留原有附件数据，不触发清理 |
| 非邮件消费 | 手动上传、OCR识别等来源 | 保存和展示逻辑均不受影响 |
| 开关关闭 | 未开启合并模式 | 三种格式分别生成三个消费，各存对应格式附件 |

与其他功能的兼容性

| 相关功能 | 交互关系 | 处理方案 |
|-------------------------------------------------------------------------------|-------------------------------------------------------------------------------|-------------------------------------------------------------------------------|
| 邮件拉取发票 | 核心关联 | 拉取后新增优先级判断 |
| 消费详情展示 | 展示关联 | 直接展示已保存附件，不做二次判断 |
| 发票自动生成消费 | 配置关联 | 依赖"同一封邮件内的发票和行程单合并为一笔消费"开关状态 |

## **4\. 验收标准**

| 序号 | 验收项 | 验收标准 |
|-------------------------------------------------------------------------|----------------------------------------------------------------------------|-------------------------------------------------------------------------------|
| 1 | 保存优先级-pdf优先 | 同一发票包含pdf、ofd、xml三种格式时，消费中保存pdf格式附件 |
| 2 | 保存优先级-ofd优先 | 同一发票仅包含ofd和xml格式时，消费中保存ofd格式附件 |
| 3 | 保存优先级-xml兜底 | 同一发票仅包含xml格式时，消费中保存xml格式附件 |
| 4 | 合并模式限制 | 未开启"同一封邮件内的发票和行程单合并为一笔消费"时，三种格式仍分别生成三个消费，各存对应格式附件 |
| 5 | 存量消费兼容 | 历史已生成的邮件消费，附件数据不被清理，保持原有存储状态 |
| 6 | 消费详情展示 | 新拉取生成的消费，详情页展示按优先级保存的那份附件 |
| 7 | 非邮件消费不受影响 | 手动上传、OCR识别等其他来源生成的消费，保存和展示逻辑均不受影响 |

A. 典型使用场景

场景：企业员工通过邮件拉取发票后查看消费

配置状态（管理员已配置）：
- "同一封邮件内的发票和行程单合并为一笔消费"开关已开启

1. 员工邮箱收到一张包含pdf、ofd、xml三种格式的发票邮件

结果：员工无需再打开xml格式文件，发票查看体验提升。

## 1\. 需求背景

### 1.1 需求描述

允许上传税局验真返回为"部分红冲"状态的发票，增加"未红冲金额"字段供用户手工填写，系统中引用发票金额的功能改用未红冲金额。

### 1.2 需求背景
- **需求方**：有成报销内部需求团队 / 企业客户
- **现状问题**：
• 税局中仅部分红冲的蓝票（OCR原始值=7）上传有成报销时，被统一视为"已红冲"状态，禁止上传
• 员工持有部分红冲发票时，只能换票报销或走无票报销流程，体验差、效率低
• 系统无法区分"部分红冲"和"全额红冲/已红冲"，一刀切拦截
- **解决思路**：
• 将部分红冲从红冲拦截逻辑中分离，单独映射为 PartialRedFlushes 状态
• 增加"未红冲金额"字段，由用户手工填写，系统强制校验
• 引用发票金额的功能（消费明细填充、报销单汇总、预算校验、凭证生成等）改用未红冲金额

**业务流程**
```
用户上传发票 → OCR识别/验真 → 状态映射 → 部分红冲则放行，其他红冲仍拦截
↓
发票组件展示"未红冲金额"字段 → 用户手工填写 → 系统校验（大于0且小于发票金额）
↓
关联消费明细 → 金额自动填充为未红冲金额 → 报销单汇总/预算校验/凭证生成均引用未红冲金额

```

## 2\. 功能现状

发票上传功能位于有成报销 PC 端及移动端，用户可通过 OCR 识别、验真接口、发票池选取等方式上传发票。

**现有字段**：
- `invoiceStatus`：发票状态，枚举值为 Normal / RedFlushes / Cancellation
- `invoiceAmount`：发票金额（含税），OCR识别或验真后自动填充

**现有操作**：
- 上传发票后，系统自动识别发票状态
- 状态为 RedFlushes 或 Cancellation 时，抛出异常，禁止上传
- 状态为 Normal 时，允许上传，消费明细关联后金额自动填充为 `invoiceAmount`

**缺失能力**：
- 系统无法识别"部分红冲"状态（OCR原始值=7被统一映射为 RedFlushes）
- 没有"未红冲金额"字段，无法记录部分红冲发票的可用金额
- 引用发票金额的功能没有替代字段可用

### 2.1 发票状态映射逻辑
- **OCR 原始值 → 系统状态映射**：

| OCR 原始值 | 含义 | 系统状态 |
|-------------|------|------------|
| 0 | 正常 | Normal |
| 2 | 作废 | Cancellation |
| 3 | 已红冲 | RedFlushes |
| 7 | 部分红冲 | RedFlushes（与已红冲、全额红冲统一） |
| 8 | 全额红冲 | RedFlushes（与已红冲、部分红冲统一） |
- **拦截逻辑**：`InvoiceUtils.checkInvoiceIsRedAndCancel()` 和 `ReimburseInvoiceServiceImpl.parchaseInvoiceAndSave()` 中对 `RedFlushes` 统一拦截。
> 本次需求需将部分红冲(7)从 RedFlushes 中分离，映射为独立的 PartialRedFlushes 状态。   

### 2.2 发票金额引用现状
- 消费明细关联发票后，金额字段默认引用 `invoiceAmount`
- 报销单总金额计算引用各消费明细金额（即 `invoiceAmount`）
- 预算超额校验时引用消费明细金额（即 `invoiceAmount`）
- 财务凭证生成时引用消费明细金额（即 `invoiceAmount`）
- 付款金额计算引用报销单总金额（即 `invoiceAmount` 汇总）
> 本次需求需增加 `unRedFlushAmount` 字段，部分红冲发票的相关功能引用此字段替代 `invoiceAmount`。

### 3.2 详细功能描述

#### 3.2.1 发票状态映射分离

**功能概述**

将 OCR 原始值 7（部分红冲）从 RedFlushes 中分离，映射为独立的 PartialRedFlushes 状态，使系统能够区分部分红冲和其他红冲状态。

**配置入口**

无独立配置入口，为系统底层映射逻辑改造。

**映射规则**

| OCR 原始值 | 含义 | 改造前系统状态 | 改造后系统状态 |
|-------------|------|---------------------|---------------------|
| 0 | 正常 | Normal | Normal |
| 2 | 作废 | Cancellation | Cancellation |
| 3 | 已红冲 | RedFlushes | RedFlushes |
| 7 | 部分红冲 | RedFlushes | **PartialRedFlushes** |
| 8 | 全额红冲 | RedFlushes | RedFlushes |

**默认配置**

| 字段 | 默认状态 | 说明 |
|------|------------|------|
| 部分红冲映射 | PartialRedFlushes | 固定映射，不可配置 |

**配置存储** • 映射逻辑为代码层硬编码，不存储配置
• 验真接口返回的状态映射与 OCR 场景保持一致

#### 3.2.2 红冲拦截逻辑调整

**功能概述**

修改红冲检测拦截逻辑，PartialRedFlushes 状态不再拦截，仅 RedFlushes 和 Cancellation 状态拦截。

**业务流程**
```
发票上传（OCR批量 / 验真单张）
↓
状态映射（convertInvoiceStatus）
↓
红冲检测（checkInvoiceIsRedAndCancel / parchaseInvoiceAndSave）
↓
Normal / PartialRedFlushes → 放行
RedFlushes / Cancellation → 拦截，抛出异常

```

**效果示例**
```
┌─────────────────────────────────────────┐
│ 发票上传结果                            │
├─────────────────────────────────────────┤
│ 正常发票：3张    ✅ 上传成功             │
│ 部分红冲：1张    ✅ 上传成功（新增）     │
│ 已红冲：1张      ❌ 已冲红，禁止上传!   │
│ 作废：1张        ❌ 已作废，禁止上传!   │
└─────────────────────────────────────────┘

```

#### 3.2.3 未红冲金额字段

**功能概述**

发票组件新增"未红冲金额"字段，仅对部分红冲发票显示，用户必须手工填写，系统强制校验。

**配置入口**

有成报销 → 单据填写页 → 消费明细 → 发票组件

**配置弹窗/界面**

```
│ │    后使用                              │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘

```

**默认配置**

| 字段 | 默认状态 | 说明 |
|------|------------|------|
| 未红冲金额 | 空 | 用户必须手工填写，无默认值 |
| 显示条件 | 仅 PartialRedFlushes | 非部分红冲发票不显示该字段 |

**配置存储** • 未红冲金额存储于当前单据关联的发票记录中
• 不反向写入发票池原始数据

**校验规则**

| 校验项 | 规则 | 错误提示 |
|---------|------|------------|
| 必填 | 不能为空 | "请填写未红冲金额" |
| 大于0 | 值 \> 0 | "未红冲金额必须大于0" |
| 小于发票金额 | 值 \< invoiceAmount | "未红冲金额必须小于发票金额" |

**交互规则**
- 未红冲金额填写后支持修改，修改时重新触发校验
- 校验未通过时，无法关联消费明细或提交单据
- 非部分红冲发票不显示该字段，保持原逻辑

#### 3.2.4 发票金额引用替换

**功能概述**

当发票状态为 PartialRedFlushes 且 unRedFlushAmount 已填写时，系统中所有原引用 invoiceAmount 的功能改用 unRedFlushAmount。

**业务流程**
```
用户填写未红冲金额（如：发票金额1000，填写700）
↓
关联消费明细 → 消费明细金额自动填充为 700
↓
报销单金额汇总 → 按 700 汇总
↓
预算超额校验 → 按 700 校验
↓
财务凭证生成 → 按 700 生成
↓
付款金额计算 → 按 700 计算

```

**效果示例**
```
消费明细
┌──────────┬──────────┬──────────┐
│ 消费事由  │ 摘要      │ 金额      │
├──────────┼──────────┼──────────┤
│ 住宿费    │ 郑州酒店  │ CNY 700.00│  ← 未红冲金额
└──────────┴──────────┴──────────┘

报销单汇总
总金额：CNY 700.00

```

**主要影响点**

| 功能模块 | 改造前引用 | 改造后引用（部分红冲时） |
|------------|---------------|------------------------------------|
| 消费明细金额自动填充 | invoiceAmount | unRedFlushAmount |
| 报销单金额汇总 | invoiceAmount | unRedFlushAmount |
| 预算超额校验 | invoiceAmount | unRedFlushAmount |
| 财务凭证生成 | invoiceAmount | unRedFlushAmount |
| 发票统计报表 | invoiceAmount | unRedFlushAmount |
| 付款金额计算 | invoiceAmount | unRedFlushAmount |

**兜底规则** • 若 unRedFlushAmount 为空，消费明细金额不自动填充，阻止提交
• 非部分红冲发票保持引用 invoiceAmount

#### 3.2.5 发票卡片提示

**功能概述**

部分红冲发票卡片增加状态标签和底部橙色提示条，提醒用户该发票已部分红冲。

**展示内容**
- 发票状态标签：显示"部分红冲"
- 卡片底部橙色提示条："该发票已部分红冲，请确认未红冲金额后使用"

**样式规则**
- 提示条样式与现有"发票开票时间超过4个月"等提示条一致
- 橙色背景、感叹号图标、单行文字

#### 3.2.6 单据详情页提示

**功能概述**

单据详情页顶部区域增加橙色提示文字，提醒审批人/查看人该单据包含部分红冲发票。

**展示内容**
- 提示文字："存在部分红冲的发票"

**展示规则**
- 仅当单据中包含至少一张 PartialRedFlushes 状态的发票时显示
- 提示文字为橙色，不影响页面主要操作

### 3.3 逻辑互斥与边界处理

**边界情况处理**

| 边界场景 | 触发条件 | 处理方案 |
|------------|------------|------------|
| 未红冲金额为空 | 用户未填写未红冲金额 | 阻止关联消费明细，阻止提交单据，提示"请填写未红冲金额" |
| 未红冲金额为0 | 用户填写0或负数 | 校验失败，提示"未红冲金额必须大于0" |
| 未红冲金额≥发票金额 | 用户填写等于或大于发票金额 | 校验失败，提示"未红冲金额必须小于发票金额" |
| 未红冲金额修改 | 用户修改已填写的未红冲金额 | 重新触发校验，校验通过后更新所有引用 |
| 多张部分红冲发票 | 单据中包含多张部分红冲发票 | 每张发票独立填写未红冲金额，独立校验 |
| 混合发票类型 | 单据中同时包含部分红冲和正常发票 | 部分红冲按 unRedFlushAmount 计算，正常发票按 invoiceAmount 计算 |
| 存量单据兼容 | 改造前已提交的单据 | 改造前部分红冲发票无法上传，不存在存量单据场景 |
| 发票池展示 | 用户在发票池中查看部分红冲发票 | 显示"部分红冲"状态，可正常查看和管理 |

**与其他功能的兼容性**

| 相关功能 | 交互关系 | 处理方案 |
|------------|------------|------------|
| 预算超额校验 | 关联操作 | 部分红冲发票按 unRedFlushAmount 参与预算校验 |
| 财务凭证生成 | 关联操作 | 部分红冲发票按 unRedFlushAmount 生成凭证金额 |
| 发票验真 | 前置操作 | 验真返回部分红冲状态时放行，不再拦截 |
| 发票池管理 | 独立操作 | 发票池中正常展示部分红冲发票及状态 |
| 移动端上传 | 独立操作 | 移动端同步支持部分红冲发票上传和未红冲金额填写 |

## 附录

**A. 典型使用场景**

场景：员工持有部分红冲的住宿发票，需要在有成报销中提交报销单

配置步骤（首次）：
1. 无需额外配置，系统底层映射逻辑改造后自动生效

使用步骤（日常）：
1. 员工在报销单中点击"添加发票"
2. 上传部分红冲的住宿发票图片
3. OCR 识别后，系统识别为"部分红冲"状态，发票卡片显示"未红冲金额"字段
4. 员工查看发票原始金额（如 1000 元），根据实际未红冲金额填写（如 700 元）

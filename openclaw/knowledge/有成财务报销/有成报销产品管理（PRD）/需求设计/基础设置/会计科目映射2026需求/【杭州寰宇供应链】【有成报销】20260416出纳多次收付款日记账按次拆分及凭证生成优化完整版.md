## 1\. 需求背景

### 1.1 需求描述

当前有成报销在多次收付款场景下（付款计划组件启用后），后端 `reimburse_order_payment_details` 表已记录每笔分次支付/收款的明细时间和金额，但多次支付的单据目前只生成**一条日记账**（日期通常为单据完成时间），导致后续通过日记账生成凭证时，无法反映每次实际收付款的真实日期，也无法与银行流水逐笔对应。

本次需求实现两方面优化：
1. **日记账拆分优化**：同一单据的多次出纳支付/收款，按每次实际收付日期和金额**拆分为多条独立日记账**
2. **凭证生成优化**：用户勾选日记账后，**按产品原有逻辑生成凭证**，由于日记账已按批次拆分，生成的凭证自然对应不同日期和金额，实现"一单次一凭证"的效果

### 1.2 需求背景
- **需求方**：财务人员/会计/审计人员
- **现状问题**：
• 日记账日期取完成时间，与真实支付/收款日期存在偏差，影响财务对账和审计
• 一笔单据多次支付只生成一条日记账，后续凭证无法与银行流水逐笔对应
- **解决思路**：
• 日记账生成时按支付/收款批次拆分为多条，每条日记账日期对应该批次实际收付日期
• 凭证仍通过日记账按原有逻辑生成，不直接改动单据→凭证的底层生成路径

**业务流程**
```plain
单据审批完成 → 出纳/签收按批次支付/收款
↓
每次支付写入 reimburse_order_payment_details
↓
单据完成 → 按未生成日记账的支付/收款批次拆分为多条日记账
↓
用户在日记账模块勾选日记账 → 按原有逻辑生成凭证
↓
每张凭证的日期 = 对应日记账日期 = 该批次实际支付/收款日期

```

## 2\. 功能现状

### 2.1 数据存储现状

| 存储位置 | 字段 | 当前用途 |
|------------|------|------------|
| `reimburse&#95;order&#95;payment&#95;details` | `created` | 记录每次支付/收款的实际时间，作为日记账日期拆分的数据源 |
| `reimburse&#95;order&#95;payment&#95;details` | `money` | 记录每次支付/收款的实际金额 |
| `order&#95;info` | `complete&#95;time` | 当前日记账及凭证日期常取的值 |

### 2.2 日记账生成现状
- 一张单据 → 生成 **一条日记账**
- 多次支付时，日记账金额为单据总金额，不因分次支付拆分
- 日记账日期通常取单据完成时间或固定规则，不反映每次实际收付时间

### 2.3 多次收付款现状
- **付款侧**：付款计划组件启用后，报销单/借款单/付款单支持分批次支付
- **收款侧**：收款单支持分批次签收收款
- **退款侧**：收款单退款时关联原单据，自动处理预算释放和核销

### 2.4 多次收付款产品配置

当前系统内单据是否支持多次打款，由以下产品配置控制：

| 配置项 | 配置路径 | 作用 |
|---------|------------|------|
| **多次打款启用** | 基础设置 → 自定义单据 → 设置 → 多次打款启用 | 开关开启后，对应单据类型在出纳节点支持分多次支付 |
> 说明：上述配置在本次需求中**保持原有逻辑不变**，仅作为功能前置条件在 PRD 中明确。   

### 2.5 中转凭证配置现状

当前系统内已存在中转凭证的产品配置：

| 配置项 | 配置路径 | 作用 |
|---------|------------|------|
| **参与凭证的中转单据设置** | 基础设置 → 会计科目映射 → 基础设置 → 中转凭证 → 参与凭证的中转单据设置启用 | 开关开启后，符合设置条件的单据在生成凭证时，将自动生成中转凭证 |
| **凭证中转设置** | 同一配置页面 | 可选择生成"两笔凭证"还是"一笔凭证" |

**两笔凭证 vs 一笔凭证说明**：
- **两笔凭证**：生成中转凭证（如往来科目）\+ 实际凭证两张独立凭证
- **一笔凭证**：将中转逻辑合并为一张凭证展示
> 本次需求不改变中转凭证配置逻辑。日记账生成凭证时，原有中转规则继续生效。   

### 2.6 凭证生成触发方式及代码现状（acpx 分析结论）

基于代码实际分析（acpx），当前凭证生成的关键逻辑如下：

**1\. 凭证生成是用户手动触发的，非付款操作自动触发**
- 实际触发方式：用户在"会计科目映射"页面点击"预览/生成凭证"
- 付款完成本身**不会自动**触发生成凭证

**2\. 自动生成凭证的触发条件**
- `autoGenerate` 调用（位于 `finishOrder` 中）仅在"自动生成凭证"配置开关打开时才会执行
- 若未开启自动制单，则完全依赖用户手动点击生成

**3\. 当前代码对分次支付及中转凭证的限制**
- **不支持分次付款分别生成中转凭证**
- 一张付款单在当前代码中**最多生成 2 张凭证**（1 张中转凭证 \+ 1 张正式凭证）
- **中转凭证只能生成一次**，不存在按支付批次多次生成中转凭证的能力
> 本次需求**不直接突破**单据生成凭证的代码限制，而是改为：单据先按批次拆分生成日记账，再由日记账按原有逻辑生成凭证，从而绕过"一张单据最多 2 张凭证"和"中转凭证只能生成一次"的限制。   

### 2.7 单据生成凭证 vs 日记账生成凭证

当前系统中，这是两套独立逻辑：
- **单据生成凭证**：会计科目映射 → 选单据 → 生成凭证（**本次不改动**）
- **日记账生成凭证**：日记账模块 → 勾选日记账 → 按日记账现有逻辑生成凭证（**本次核心路径**）
> 本次需求的核心路径是"单据 → 日记账 → 凭证"，通过拆分日记账实现最终凭证的日期和金额与每次收付一一对应。

### 3.1 功能变更清单

| 序号 | 变更类型 | 功能点 | 功能说明 | 优先级 |
|------|------------|---------|------------|---------|
| 1 | 优化 | 日记账按每次收付款拆分生成 | 同一单据的多次支付/收款，按批次拆分为多条独立日记账 | P0 |

### 3.2 详细功能描述

#### 3.2.1 日记账按每次收付款拆分生成（核心变更）

**规则名称**：一单次一日记账

**触发入口**：
1. **系统自动生成**：单据完成并支付后，满足自动生成日记账条件时触发
2. **用户手动生成**：财务管理 → 凭证日记账 → 日记账管理 → 点击【生成日记账】→ 选择单据 → 确认生成日记账

**拆分逻辑**：
```plain
单据状态变为「已完成」 → 判断该单据是否有多次支付/收款记录
↓
├─ 单次支付/收款 → 生成 1 条日记账（日期 = 该次实际收付日期，金额 = 该次实际收付金额）
└─ 多次支付/收款 → 按支付/收款批次拆分为 N 条独立日记账
   ├─ 每条日记账金额 = 该批次 payment_details.money
   ├─ 每条日记账日期 = 该批次实际支付/收款时间 payment_details.created
   └─ 日记账摘要、科目、辅助核算等按原规则取值

```

**与凭证的关系**：
- 日记账生成后，用户在日记账模块勾选日记账，按**产品原有逻辑**生成凭证
- 由于日记账已按批次拆分且日期不同，生成的凭证自然对应不同日期和金额
- **单据直接生成凭证的路径（会计科目映射 → 选单据 → 生成凭证）保持不变**，本次需求不改动

**日记账内容一致性**：

| 日记账字段 | 取值规则 |
|---------------|------------|
| 日期 | 当存在多次支付时，取该批次 `payment&#95;details.created`；单次时同样取该次实际日期 |
| 金额 | 该批次实际支付/收款金额 |
| 借方/贷方科目 | 与原整单日记账一致 |
| 辅助核算 | 与原整单日记账一致 |
| 摘要 | 按产品原有摘要逻辑生成，本次需求不变更 |

#### 3.2.2 自动生成日记账适配

**自动制单设置影响**：
- 自动生成日记账的触发时机保持原逻辑不变：**仅在单据状态变更为「已完成」（ORDER\_STATUS\_6）时触发**
- 触发后，不再为整单生成一条日记账，改为按支付/收款批次拆分，**为每个未生成日记账的批次各生成一条日记账**
- 若某次支付因网络/异常未生成日记账，下次自动制单任务扫描时补生成

### 3.3 逻辑互斥与边界处理

**边界情况处理**

| **边界场景** | **触发条件** | **处理方案** |
|----------------|----------------|----------------|
| 部分支付后关闭单据 | 审批流配置允许关闭未完成单据 | 已支付批次正常生成日记账；未支付批次不再生成日记账 |
| 退款后重新收款 | 收款单退款后又重新收款 | 退款和重新收款作为独立批次，分别生成日记账 |
| 线上支付回调延迟 | 第三方支付成功回调晚于操作时间 | 日记账日期以 `payment&#95;details.created`（系统记录的支付时间）为准，而非回调时间 |
| 自动生成日记账触发时机 | 单据处于审批中、待支付等中间状态 | 保持原逻辑：仅在单据状态变为「已完成」时才触发自动生成日记账 |
| 未配置付款计划 | 单据未启用多次打款/收款 | 按单次处理，生成一条日记账 |
| 某批次已生成日记账 | 重复触发或自动任务重试 | 已生成日记账的批次置灰/标记已生成，不再重复生成 |
| 外币支付 | 存在汇率换算 | 每次日记账按该批次外币金额及汇率独立计算本币金额 |
| 凭证作废 | 用户在凭证管理模块作废凭证 | 作废某张凭证不影响同单据关联的其他日记账和其他凭证 |
| 中转凭证启用 | 日记账生成凭证时启用了中转 | 按原有日记账生成凭证逻辑执行，中转规则（两笔/一笔）生效 |

**与其他功能的兼容性**

| **相关功能** | **交互关系** | **处理方案** |
|----------------|----------------|----------------|
| 预算释放 | 支付后释放 | 按原逻辑执行，不因日记账拆分而改变 |
| 凭证推送（财务系统） | 外部对接 | 多张凭证逐张推送，外部系统按凭证号分别接收 |
| 项目报表 | 数据统计 | 项目报表中单据金额仍按总金额统计，不因日记账拆分而重复 |
| 单据直接生成凭证 | 会计科目映射路径 | 保持原逻辑不变，本次不改动 |

## 5\. 验收标准

| **序号** | **验收项** | **验收标准** |
|----------|-------------|----------------|
| 1 | 单次支付日记账日期 | 单次支付的单据，生成的日记账日期取 `payment&#95;details.created`，可与 `completeTime` 不同 |
| 2 | 日记账按次拆分 | 同一单据分 3 次支付，生成 3 条独立日记账，每条金额等于对应批次金额 |
| 3 | 日记账生成凭证 | 勾选上述 3 条日记账按原有逻辑生成凭证，产生 3 张不同日期的凭证 |
| 4 | 自动制单适配 | 启用自动生成日记账后，单据状态变为「已完成」时自动按批次拆分生成日记账 |
| 5 | 收款单同步 | 收款单按收款日期拆分生成日记账 |
| 6 | 未拆分场景兼容 | 未启用多次收付款的单据，仍按单次生成 1 条日记账 |
| 7 | 部分支付后关闭 | 已支付批次生成日记账，关闭后未支付批次不再生成日记账 |
| 8 | 重复生成拦截 | 已生成日记账的支付/收款批次，再次触发时置灰或标记已生成 |
| 9 | 单据直接生成凭证路径 | 会计科目映射 → 选单据 → 生成凭证的逻辑保持不变 |

## 附录

**A. 典型使用场景**

**场景**：报销单总额 10,000 元，分两次支付给供应商
- 3 月 1 日支付 3,000 元（首付款）
- 3 月 15 日支付 7,000 元（尾款）

**变更前**：
- 生成 1 条日记账，日期为完成时间（如 3 月 20 日），金额 10,000 元
- 由日记账生成 1 张凭证，日期 3 月 20 日，金额 10,000 元

**变更后**：
- 自动生成 2 条日记账： 
    - 日记账 1：金额 3,000 元，日期 3 月 1 日
    - 日记账 2：金额 7,000 元，日期 3 月 15 日
- 用户在日记账模块勾选 2 条日记账，按原有逻辑生成 2 张凭证： 
    - 凭证 1：金额 3,000 元，日期 3 月 1 日
    - 凭证 2：金额 7,000 元，日期 3 月 15 日
- 两张凭证号连续，均可与银行流水逐笔对账

**B. 字段说明**

| 字段 | 数据来源 | 用途 |
|------|------------|------|
| `reimburse&#95;order&#95;payment&#95;details.created` | 支付/收款操作时刻 | 拆分日记账时的日期取值 |
| `reimburse&#95;order&#95;payment&#95;details.money` | 本次收付金额 | 拆分日记账时的金额 |
| `order&#95;info.complete&#95;time` | 单据完成时间 | 未配置多次收付款时的 fallback 日期 |

**C. 代码注意事项**
```java
// 日记账拆分核心变更点：
// 1. 查询单据关联的全部 payment_details 记录
// 2. 过滤掉已生成日记账的批次
// 3. 对剩余批次逐一生成日记账记录
// 4. 日记账日期 = paymentDetail.getCreated()
// 5. 日记账金额 = paymentDetail.getMoney()
// 6. 凭证仍由日记账按原有逻辑生成，不改动单据直接生成凭证路径

```

## 〇、现有逻辑说明（供参考）

### 0.1 预算占用机制现状
- **报销单/付款单/借款单**：提交送审时按预算维度写入 `order_budget_info`（预算占用记录表，status=1 占用），审批通过→status=2（使用），驳回/撤回→status=3（释放）
- **超额校验公式**：`剩余 = budgetMoney（预算总额） - usedMoney（已使用金额） - occupiedMoney（占用中金额）`（基于 `BudgetRuleDetail` 的年度\+周期粒度）
- **预算申请单**：目前无冻结机制，提交后直接进入审批流，变更额度在审批期间可被其他单据占用

### 0.2 预算体系架构现状
- **BudgetRuleInfo**：`code`（维度编码）/`parentCode`（父维度编码）树形结构，`dimensionType`（维度类型）=1提交人/2部门/3费用类型/4项目
- **BudgetRuleDetail**：按 `budgetYear`（预算年度）\+ `cycleRank`（周期排序，如1=1月/2=2月...）拆分预算明细，是实际占用/校验的最小粒度
- **BudgetControlInfo**：仅存储控制策略（`excessOperation` 等），不直接存储周期明细金额
- **预算申请核心方法**：`BudgetApplyService.updateBudgetByApplyOrder()`，逐维度遍历（每个 `ApplyDetail`（预算申请明细）对应一个 `code`（维度编码）），逐周期更新（`CycleDetail`（周期明细）的 `oldAmount/newAmount`）
- **关键发现**：当前系统**不支持跨维度调整**。`ApplyDetail`（预算申请明细）只有单个 `code`（维度编码）字段，无 `targetCode/sourceCode`（目标维度编码/来源维度编码）等跨维度字段
- **动态预算**：`DynamicBudgetConfigPO`（动态预算配置表）通过公式异步计算 `budgetMoney`（预算总额），本次需求暂不覆盖

### 0.3 超额策略现状

| 数据库值 | 含义 | 行为 |
|------------|------|------|
| 0 | 提示（弱控制） | 超标但允许提交 |
| 1 | 禁止（强控制） | 超标拦截，不可提交 |
| 2 | 不提示 | 超标无感 |
> 注意：代码中 `excessOperation`（超额操作类型）存储值为 0/1/2，前端展示映射为提示/禁止/不提示。   

### 0.4 预算申请现有能力（代码分析结论）

| 能力 | 说明 | 本次需求是否涉及冻结 |
|------|------|------------------------------|
| **追加模式**（budgetApplyFillType=1） | 新旧金额差额叠加到现有预算（`budgetMoney += (newAmount - oldAmount)`，newAmount=申请金额, oldAmount=原金额） | ❌ 不冻结（增加预算，不影响当前可用余额） |
| **修改模式-金额上调**（fillType=0, newAmount \> oldAmount） | 直接替换为新金额（`budgetMoney = newAmount`，newAmount=申请后的新金额） | ❌ 不冻结 |
| **修改模式-金额下调/调减**（fillType=0, newAmount \< oldAmount） | 直接替换为新金额（`budgetMoney = newAmount`，newAmount=申请后的新金额，预算减少） | ✅ **需要冻结**（防止审批期间额度被继续消耗） |
| **新增维度**（newRule=true） | 新增一个预算规则维度 | ❌ 不冻结（无现有额度） |
| **历史周期修改管控**（allowModifyHistory） | 配置是否允许修改历史周期 | ❌ 与冻结无关 |
> **结论**：当前系统**不支持跨维度调整**。冻结机制仅针对"同维度金额下调（调减）"。

## 一、快速概览

### 1.1 这个需求要做什么？

**一句话**：预算申请单提交审批期间，对"同维度金额下调（调减）"涉及的预算周期额度进行冻结，防止审批期间其他单据超额使用。

**具体效果**：
- 同维度调减提交后，目标周期额度被冻结，其他单据无法继续消耗
- 审批通过后冻结金额生效并释放，驳回/撤回后释放冻结
- 调减生效后自动兜底校验，标记超标占用单据
- 预算看板可查看「冻结中」额度
- 追加/金额上调/新增维度不冻结
> **追加/上调不冻结的理由**：增加预算不影响当前可用余额。若冻结当前余额，反而让预算更紧张，与业务意图矛盾。   

### 1.2 谁需要这个功能？

| 角色 | 痛点 | 解决方案 |
|------|------|------------|
| 企业财务管理员 | 预算调减审批期间，报销单继续提交导致调减后实际超支 | 调减提交即冻结目标周期额度，审批通过前其他单据无法占用 |
| 审批人 | 审批通过后发现预算已超，需手动追回转审 | 调减生效前即保护剩余额度，审批人无需事后追回转审 |

### 1.3 核心价值
- ✅ 调减审批期间预算不被"透支"，预算控制实时准确
- ✅ 调减生效后自动识别超标单据，减少财务人工排查
- ✅ 冻结粒度落到周期维度（年度\+周期），不影响其他周期单据

## 二、功能需求

### 2.1 数据模型变更

#### 2.1.1 BudgetRuleDetail 扩展冻结字段

在现有 `BudgetRuleDetail`（按 `budgetYear + cycleRank` 拆分）上新增冻结金额字段：

| 新增字段 | 类型 | 说明 |
|------------|------|------|
| `frozenMoney` | decimal(16,2) | 该周期内的冻结金额（汇总该周期下所有审批中调减申请单的冻结差额） |
> **设计说明**：冻结落到 `BudgetRuleDetail`（预算规则明细表）周期粒度，而非 `BudgetControlInfo`（预算控制信息表）维度粒度。确保5月冻结不影响6月预算。   

#### 2.1.2 新增关联表：budget\_freeze\_record（预算冻结记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| budgetId | bigint | 预算控制ID（关联 BudgetControlInfo） |
| ruleCode | varchar(64) | 预算规则编码（关联 BudgetRuleInfo.code，叶子节点） |
| budgetYear | int | 预算年度 |
| cycleRank | int | 周期序号（月度1<sub>12，季度1</sub>4等） |
| applyOrderCode | varchar(64) | 预算申请单编号 |
| freezeAmount | decimal(16,2) | 冻结金额（正数，= 原金额 - 新金额，即本次调减的差额） |
| freezeType | tinyint | 1=调减（当前仅支持此类型，预留扩展） |
| status | tinyint | 1=冻结中, 2=已生效, 3=已释放 |
| createTime | datetime | 冻结时间 |
| updateTime | datetime | 状态变更时间 |
> **追加/金额上调/新增维度不生成冻结记录**。追加直接叠加预算，金额上调直接替换为新金额，均无冻结。   

### 2.2 预算申请单提交 — 冻结逻辑

#### 2.2.1 冻结规则

| 申请单类型 | 冻结行为 | frozenMoney 变化 |
|---------------|------------|------------------|
| 追加（fillType=1） | **不冻结** | 无（`budgetMoney += (newAmount - oldAmount)`，newAmount=申请金额, oldAmount=原金额，叠加差额） |
| 修改-金额上调（fillType=0, newAmount \> oldAmount） | **不冻结** | 无（`budgetMoney = newAmount`，newAmount=申请后的新金额，直接替换） |
| 修改-金额下调/调减（fillType=0, newAmount \< oldAmount） | 冻结调减差额 | 目标周期 `frozenMoney += (oldAmount - newAmount)`（oldAmount=原金额, newAmount=申请后的新金额, 差值为冻结金额） |
| 新增维度（newRule=true） | **不冻结** | 无（无现有额度，直接新增） |
> **冻结金额计算**：freezeAmount = oldAmount（原金额）- newAmount（申请后的新金额）   

#### 2.2.2 冻结流程
```java
// 步骤1：预算申请单提交送审
预算申请单提交送审
  ↓
// 步骤2：判断申请类型（仅调减触发冻结）
判定：是否为修改模式(fillType=0) 且 newAmount（申请后的新金额） < oldAmount（原金额）？
  ├─ 是（调减）→ 进入冻结流程
  └─ 否（追加/上调/新增）→ 跳过冻结，直接进入审批流
  ↓
// 步骤3：前置校验——目标周期剩余可用额度
// budgetMoney=预算总额, usedMoney=已使用金额, occupiedMoney=占用中金额, frozenMoney=冻结金额
校验：budgetMoney - usedMoney - occupiedMoney - frozenMoney >= (oldAmount - newAmount)
  ├─ 不满足 → 拦截提示
  └─ 满足 → 继续
  ↓
// 步骤4：写入冻结记录表
// budget_freeze_record.status=1 表示冻结中
// freezeAmount = oldAmount - newAmount（调减差额）
按预算维度+年度+周期写入 budget_freeze_record（预算冻结记录表，status=1 冻结中）
  ↓
// 步骤5：更新预算周期明细表的冻结金额字段
更新 BudgetRuleDetail.frozenMoney（该周期冻结金额汇总 += freezeAmount）
  ↓
// 步骤6：写入单据预算占用关联表（仅做关联，不占额度）
// order_budget_info.amount=0 表示预算申请单本身不占预算，只做单据关联
写入 order_budget_info（复用现有机制，关联申请单，amount=0 仅做关联）
  ↓
// 步骤7：触发超额校验（所有单据提交时统一公式）
校验超额（纳入 frozenMoney 冻结金额计算，校验报销单是否超过剩余可用预算）

```

#### 2.2.3 冻结失败处理

| 场景 | 处理 |
|------|------|
| 调减时目标周期剩余不足 | 前置校验拦截：`budgetMoney - usedMoney - occupiedMoney - frozenMoney &gt;= (oldAmount - newAmount)`（budgetMoney=预算总额, usedMoney=已使用金额, occupiedMoney=占用中金额, frozenMoney=冻结金额, oldAmount=原金额, newAmount=新金额），不满足则提示「当前周期剩余预算 N 元，最多可调减 M 元」 |
| 冻结写入失败 | 阻塞队列重试（复用现有 BudgetConsumerNew 机制），重试3次失败后回滚并通知申请人 |

### 2.3 审批通过 — 生效逻辑

#### 2.3.1 各类型生效行为

| 申请单类型 | 生效行为 |
|---------------|------------|
| 追加（fillType=1） | `budgetMoney += (newAmount - oldAmount)`（newAmount=申请金额, oldAmount=原金额, 叠加差额），冻结记录无 |
| 修改-金额上调（fillType=0, newAmount \> oldAmount） | `budgetMoney = newAmount`（newAmount=申请后的新金额, 直接替换），冻结记录无 |
| 修改-金额下调/调减（fillType=0, newAmount \< oldAmount） | `budgetMoney = newAmount`（newAmount=申请后的新金额, 替换为新金额），`frozenMoney -= freezeAmount`（释放冻结金额），freeze\_record.status=2（已生效），触发兜底校验 |
| 新增维度（newRule=true） | 新增 BudgetRuleInfo \+ BudgetRuleDetail，冻结记录无 |
> **冻结释放时机**：调减审批通过时，先更新 `budgetMoney = newAmount`，再 `frozenMoney -= freezeAmount`，最后 `freeze_record.status=2`。   

#### 2.3.2 调减生效时的兜底校验（关键）
```java
// 步骤1：调减审批通过，预算总额替换为新金额，冻结金额释放
调减生效后
  ↓
// 步骤2：扫描该周期下所有审批中的报销单
// order_budget_info.status=1 表示占用中（审批中的报销单/付款单/借款单）
检查该周期下所有 status=1（占用中）的报销单
  ↓
// 步骤3：重新计算该周期累计占用金额
// totalOccupied = 所有占用中单据的 amount（占用金额）之和
totalOccupied = Σ 该周期下所有占用中单据的 amount（占用金额）
  ↓
// 步骤4：判定累计占用是否超过调减后的可用额度
// budgetMoney=新预算总额(已替换), usedMoney=已使用金额, frozenMoney=冻结金额(已释放后)
判定：totalOccupied（累计占用金额） > (budgetMoney - usedMoney - frozenMoney)？
  ├─ 是 → 标记该周期下全部占用单据为"预算超标"
  └─ 否 → 无操作
  ↓
// 步骤5：按企业配置的超额策略处理超标单据
// excessOperation: 0=提示(弱控制), 1=禁止(强控制), 2=不提示
按超额策略处理超标单据：
  强控制（excessOperation=1 禁止）→ 在审批流最终节点前拦截
  弱控制（excessOperation=0 提示）→ 标记超标，允许继续但提示审批人
  不提示（excessOperation=2 不提示）→ 仅标记，无感

```
> **判定逻辑修正**：按"累计占用"判定，而非逐张单据判定。`occupiedMoney`（占用中金额）是累计值，不能逐张扣除自身后比较。**兜底校验覆盖的两种核心场景**：   

### 2.4 驳回/撤回 — 释放冻结

| 操作 | 处理 |
|------|------|
| 驳回 | `frozenMoney -= freezeAmount`（释放冻结金额），freeze\_record.status=3（已释放） |
| 撤回 | `frozenMoney -= freezeAmount`（释放冻结金额），freeze\_record.status=3（已释放） |
| 超时未审批（30天） | 系统自动释放：`frozenMoney -= freezeAmount`（释放冻结金额），freeze\_record.status=3（已释放），通知申请人和预算管理员 |

### 2.5 冻结后的超额校验（统一公式）

**所有单据提交时的超额校验公式更新为**：
```java
// 超额校验公式（所有单据统一使用）
// budgetMoney: 预算总额（该周期配置金额）
// usedMoney: 已使用金额（审批通过的单据累计）
// occupiedMoney: 占用中金额（审批中的报销单/付款单/借款单累计）
// frozenMoney: 冻结金额（审批中的调减申请单累计）
剩余可用 = BudgetRuleDetail.budgetMoney
         - usedMoney
         - occupiedMoney
         - frozenMoney

```
> 追加/金额上调审批通过后，`budgetMoney`（预算总额）增加，后续单据校验时自然享受新增额度。   

### 2.6 审批条件中的预算判断

**审批条件判断公式更新为**：
```java
// 审批条件预算判断公式
// usedMoney: 已使用金额
// occupiedMoney: 占用中金额
// frozenMoney: 冻结金额（新增纳入）
// budgetMoney: 预算总额
预算使用率 = (usedMoney + occupiedMoney + frozenMoney) / budgetMoney

```
> 与 `BudgetConditionHandleService`（审批条件处理服务）联动，确保审批条件分支在预算冻结场景下正确触发。

## 三、边界场景处理

### 3.1 调减时目标周期剩余不足

**场景**：目标周期 budgetMoney=10000（预算总额），usedMoney=8000（已使用金额），occupiedMoney=0（占用中金额），frozenMoney=1000（冻结金额），申请从10000调减到7000（freezeAmount=3000，调减差额=原金额-新金额）   **前置校验**：10000 - 8000 - 0 - 1000 = 1000 \< 3000   **处理**：提交时拦截，提示「当前周期剩余预算 1000 元，最多可调减 1000 元（即最低可设为 9000）」

### 3.2 同周期多张申请单同时审批

**场景**：A用户和B用户同时提交调减同一周期   **处理**：通过 `BlockingQueue`（阻塞队列，用于并发请求串行处理）串行消费 \+ `BudgetRuleDetail` 行锁，按提交时间排队处理，后到的申请单提交时重新校验剩余额度（含先到的冻结）

### 3.3 冻结期间预算又被其他申请单调减

**场景**：周期X已有冻结1000（A用户调减），此时B用户又提交调减500   **处理**：后到的申请单提交时，`frozenMoney` 已包含先到冻结，校验公式自然扣减，无需特殊处理

### 3.4 审批中修改申请单金额

**场景**：用户撤回修改重提，原冻结1000（原金额10000→新金额9000），新改为冻结800（原金额10000→新金额9200）   **处理**：原子替换流程——先冻结新差额（800），冻结成功后再释放旧冻结（1000）。若新差额冻结失败，保持旧冻结不变，提示用户修改金额过大

### 3.5 调减生效后追加申请单通过

**场景**：周期X先调减1000（budgetMoney从10000→9000），后追加500（budgetMoney从9000→9500）   **处理**：调减审批通过时触发兜底校验（按9000判定）；追加审批通过时 budgetMoney 叠加至9500，不触发兜底校验

### 3.6 预算申请单30天未审批

**场景**：调减申请单提交后长期无人审批   **处理**：系统自动释放冻结（`frozenMoney -= freezeAmount` 释放冻结金额，`status=3` 已释放），通知申请人和预算管理员

### 3.7 动态预算维度提交调减

**场景**：企业启用了动态预算，对该预算维度提交调减   **处理**：系统提示「动态预算暂不支持冻结，请关闭动态预算后再操作」或直接拦截提交

### 3.8 父子维度冻结传递

**场景**：冻结了子维度（如「部门A-项目X」），父维度（「部门A」）汇总计算   **处理**：
- `frozenMoney` 记录在叶子节点（最细粒度维度）的 `BudgetRuleDetail` 上
- 父维度汇总时，`selfFrozenMoney`（父维度自身直接冻结金额）仅含自身直接冻结，`totalFrozenMoney`（父维度汇总冻结金额，含所有子维度冻结汇总）含所有子维度冻结汇总
- 超额校验和审批条件判断使用 `totalFrozenMoney`（汇总冻结金额）

### 3.9 先占用预算单据，后调减预算（场景A）

**场景**：报销单先提交占用预算，预算申请调减单后提交并审批通过   **示例**：
1. 周期X budgetMoney（预算总额）= 10000，usedMoney（已使用金额）= 6000，occupiedMoney（占用中金额）= 0
2. 报销单A提交（占用2000）→ occupiedMoney = 2000，剩余可用 = 10000 - 6000 - 2000 = 2000
3. 预算申请调减单提交（10000→7000，freezeAmount = 3000）→ frozenMoney = 3000
4. 调减审批通过 → budgetMoney = 7000，frozenMoney = 0
5. **兜底校验**：totalOccupied（累计占用金额）= 2000 \> (7000 - 6000 - 0) = 1000 → **超标**
6. 报销单A被标记为"预算超标"，按超额策略处理

**核心问题**：报销单提交时预算是够的，但调减后预算变少了，导致已有占用超标。兜底校验必须在调减生效时统一扫描处理。

### 3.10 调减审批中，新单据提交（场景B）

**场景**：预算申请调减单已提交（冻结中），期间有新报销单提交   **示例**：
1. 周期X budgetMoney = 10000，usedMoney = 6000，occupiedMoney = 0，frozenMoney = 0
2. 预算申请调减单提交（10000→8000，freezeAmount = 2000）→ frozenMoney = 2000
3. 报销单B提交（申请占用3000）： 
    - 前置校验：剩余 = 10000 - 6000 - 0 - 2000 = 2000 \< 3000 → **拦截**（新单据无法提交）
4. 假设报销单B只申请占用1500： 
    - 前置校验：剩余 = 10000 - 6000 - 0 - 2000 = 2000 \>= 1500 → **通过**
    - occupiedMoney = 1500，此时总占用 = 6000 \+ 1500 = 7500 \< 10000（原预算）
5. 调减审批通过 → budgetMoney = 8000，frozenMoney = 0
6. **兜底校验**：totalOccupied = 1500 \> (8000 - 6000 - 0) = 2000？**不超标**（1500 \< 2000，正常）

**变体 — 临界超标**：
1. 同上，调减单冻结2000（10000→8000）
2. 报销单C提交（申请占用2500）： 
    - 前置校验：剩余 = 10000 - 6000 - 0 - 2000 = 2000 \< 2500 → **拦截**（无法提交）
3. 假设企业关闭了冻结功能或校验有漏洞，报销单C成功提交占用2500
4. 调减审批通过 → budgetMoney = 8000
5. **兜底校验**：totalOccupied = 2500 \> (8000 - 6000) = 2000 → **超标**
6. 报销单C被标记为"预算超标"，按超额策略拦截或提示

**核心问题**：调减期间新单据提交时，`frozenMoney` 已扣减，校验看似通过；但调减后 `budgetMoney` 下降，可能导致单据实际超标。兜底校验作为最后一道防线，统一扫描处理。

## 五、附录

### 5.1 典型使用场景

**场景1：调减预算导致周期超标**
1. 维度X-2026年5月预算总额 10000，已用 8000，冻结 0
2. 提交调减 3000（原10000→新7000）→ 前置校验：10000 - 8000 - 0 - 0 = 2000 \< 3000 → 拦截
3. 管理员改为调减 1500（原10000→新8500）→ 校验通过
4. 审批通过 → budgetMoney（预算总额）= 8500，frozenMoney（冻结金额）= 0
5. 兜底校验：累计占用 8000 \<= 8500 → 无超标

**场景2：追加后调减**
1. 维度X-2026年5月预算总额 10000，已用 8000
2. 提交追加 2000（fillType=1，原10000→新12000）→ 不冻结
3. 追加审批通过 → budgetMoney（预算总额）= 12000
4. 同月又提交调减 3000（修改模式，原12000→新9000，oldAmount=12000, newAmount=9000）→ 冻结3000
5. 调减审批通过 → budgetMoney（预算总额）= 9000，frozenMoney（冻结金额）= 0
6. 兜底校验：累计占用 8000 \<= 9000 → 无超标

**场景3：先占用后调减导致超标（场景A）**
1. 维度X-2026年5月 budgetMoney（预算总额）= 10000，usedMoney（已使用金额）= 6000，occupiedMoney（占用中金额）= 0，frozenMoney（冻结金额）= 0
2. 报销单A提交（占用2000）→ occupiedMoney = 2000，剩余可用 = 10000 - 6000 - 2000 - 0 = 2000
3. 预算申请调减单提交（10000→7000，freezeAmount = 3000）→ frozenMoney = 3000
4. 调减审批通过 → budgetMoney（预算总额）= 7000，frozenMoney（冻结金额）= 0
5. 兜底校验：totalOccupied（累计占用金额）= 2000 \> (7000 - 6000 - 0) = 1000 → **超标**
6. 报销单A被标记"预算超标"，按企业配置的超额策略处理（强控制则审批流拦截）

**场景4：调减期间新单据正常流转（场景B，不超标）**
1. 维度X-2026年5月 budgetMoney = 10000，usedMoney = 6000，occupiedMoney = 0，frozenMoney = 0
2. 预算申请调减单提交（10000→8000，freezeAmount = 2000）→ frozenMoney = 2000
3. 报销单B提交（申请占用1500）： 
    - 前置校验：10000 - 6000 - 0 - 2000 = 2000 \>= 1500 → **通过**
    - occupiedMoney = 1500
4. 调减审批通过 → budgetMoney = 8000，frozenMoney = 0
5. 兜底校验：totalOccupied = 1500 \<= (8000 - 6000 - 0) = 2000 → **不超标**
6. 报销单B正常审批流转

**场景5：调减期间新单据临界超标（场景B变体）**
1. 同上，调减单冻结2000（10000→8000）
2. 报销单C提交（申请占用2500）： 
    - 前置校验：10000 - 6000 - 0 - 2000 = 2000 \< 2500 → **拦截**
    - 系统提示"当前周期剩余预算2000元，本次申请占用2500元，超出500元"
3. 用户改为申请占用1500 → 校验通过

### 5.2 方案决策记录

| 决策项 | 结论 | 决策依据 |
|---------|------|------------|
| 追加是否冻结 | **不冻结** | 追加是叠加现有预算（budgetMoney \+= 差额），冻结当前余额会让预算更紧张 |
| 金额上调是否冻结 | **不冻结** | 上调是直接替换为新金额（budgetMoney=预算总额 = newAmount，newAmount=申请后的新金额），同样增加预算，不冻结 |
| 调减是否冻结 | **冻结** | 下调减少预算，需保护剩余额度不被继续消耗 |
| 跨维度调整 | **不支持** | 代码分析确认：`ApplyDetail`（预算申请明细）只有单个 `code`（维度编码），无 `targetCode/sourceCode`（目标维度编码/来源维度编码）字段，`updateBudgetByApplyOrder()`（预算申请更新方法）只操作同一维度 |
| 冻结粒度 | **BudgetRuleDetail 周期粒度** | 与现有占用/校验体系保持一致，避免维度级冻结误伤其他周期 |
| 超标判定逻辑 | **累计占用判定** | 与现有 occupiedMoney 汇总逻辑一致，避免逐张扣除自身的重复计算问题 |
| 动态预算 | **本次排除** | 动态预算 budgetMoney 由公式异步计算，冻结机制需联动公式引擎，复杂度超出本期范围 |
| 超时释放 | **30天自动释放** | 防止长期审批导致预算长期不可用，释放后通知相关方 |

## 1. 需求背景

### 1.1 需求描述

建立有成报销与携程商旅的组织架构及人员部门自动同步更新机制，通过同步接口返回的错误信息识别ID冲突问题，并引导用户解决后重新同步，同步成功后锁定开关不可关闭。

### 1.2 需求背景

- **需求方**：有成报销生态客户（使用携程商旅服务的企业客户）
- **现状**：客户先开通了携程且有组织架构，后开通了有成报销中的携程商旅，有成报销组织架构内人员信息变更后，携程侧组织架构内人员未同步变更。目前有成报销是将组织架构内所有人员信息同步至携程侧组织架构下的默认部门内，客户在携程侧后台手动做人员调整。
- **关键限制**：携程侧组织架构人事信息调整接口仅允许同一个员工在一个部门下。

**三种场景**：

| 场景 | 说明 | 处理方式 |
|------|------|----------|
| 场景一：先开通携程，后开通有成（ID冲突） | 客户先开通携程商旅，携程侧已创建组织架构（组织ID 001），后开通有成报销（组织ID 002），未触发携程人事同步接口 | 联系携程客服清空携程已有组织架构数据，重新触发携程人事同步接口，携程使用有成组织架构ID并更新组织内人员 |
| 场景二：存量客户 | 有成组织架构人员信息仍同步携程侧组织架构下默认部门内 | 首次同步成功后，同步开关锁定，不允许关闭 |
| 场景三：携程新客 | 先开通有成报销，后开通携程商旅 | 开通携程商旅时，有成报销通过接口将组织架构同步至携程侧，后续有成报销组织架构员工变更继续同步至携程侧 |

**业务流程**

```
用户开启同步开关，勾选同步有成报销完整部门人员信息
        ↓
┌─────────────────┐
│  调用携程同步接口  │
└────────┬────────┘
         ↓
    ┌────┴────┐
   成功       失败
    ↓          ↓
 同步完成   解析错误信息
    ↓          ↓
开关保持开启 是否ID冲突？
            ┌──┴──┐
           是      否
            ↓      ↓
      引导清空数据  显示通用错误
            ↓      ↓
      清空后重试   联系客服
```

## 2. 功能现状

**功能位置**：应用中心 → 携程商旅 → 员工配置 → 同步设置

**现有能力**：
- 有成组织架构内员工信息自动同步至携程商旅下默认部门；走定时任务触发同步接口
- 员工同步可手动触发同步，将有成组织架构内员工信息同步至携程商旅下默认部门
- 过滤同步：仅同步内部应用已授权的员工信息；否则默认全部同步
- 黑名单：黑名单指定的人员将不会同步到携程商旅
- 解除同步：将选中的员工信息同步至携程改变其在职状态为离职
- 携程申请单推送接口和人事同步接口会限流，有概率推送失败

**当前缺失能力**：
- 无法将有成组织架构内完整的部门和部门下的人员一起同步至携程侧
- 同步失败后的错误分类处理机制
- 基于错误类型的引导流程

### 3.2 详细功能描述

#### 3.2.1 部门与员工信息更新同步触发与错误处理流程

**功能概述**：

a. 用户勾选"同步有成报销完整部门人员信息"选项，开启"自动同步"开关点击"确定"或勾选"员工"点击"员工同步"后，系统立即调用组织架构同步和携程人事同步接口，根据接口返回结果或错误信息进行相应处理。

b. 部门与员工信息更新同步生效后，自动同步定时触发，将有成组织内全量部门信息和员工信息通过接口同步至携程走更新，不受同步状态影响。

c. 部门与员工信息更新同步生效后，"同步有成报销完整部门人员信息"选项保持已勾选置灰，不允许关闭。

d. 用户未勾选"同步有成报销完整部门人员信息"选项，开启"自动同步"开关点击"确定"或勾选"员工"点击"员工同步"后，按原逻辑仅将员工信息同步至携程。

e. 有成报销开通携程商旅后，首次将有成报销内组织架构同步至携程侧，携程侧若返回任何错误编码，都视为"组织id冲突"，引导客户联系携程客服。

**接口地址**：[携程商旅开发者平台](https://openapi.ctripbiz.com/#/serviceApi?apiId=1000133)

- 大客户：`CorpCustService/SaveCorpCustInfoList`
- 小客户：`SwitchAPI/Member/SetMembers`

> 注：调每个接口都需要 custID；需要具备异常重新调用机制，避免同步进程失败后无感知无二次触发。

**配置入口**：应用中心 → 携程商旅 → 员工管理 → 同步设置

**状态流转**

```
用户勾选"同步有成报销完整部门人员信息"并开启"自动同步"点击"确定"
或勾选"员工"点击"员工同步"
        ↓
调用携程组织架构同步接口
        ↓
┌─────────────────────────────────────────┐
│  等待接口返回（超时时间：60秒）            │
└─────────────────────────────────────────┘
        ↓
┌─────────────────┬─────────────────┐
│    返回成功      │    返回错误      │
└────────┬────────┴────────┬────────┘
         ↓                  ↓
  同步成功            解析错误码
  显示tips           
  锁定勾选项          错误码=ID冲突？
                      ┌─────────┴─────────┐
                     是                    否
                      ↓                    ↓
            显示ID冲突提示          显示通用错误提示
            引导清空数据            建议联系客服
```

#### 3.2.2 ID冲突错误识别与引导

**功能概述**：当人事同步接口返回组织架构ID不匹配错误时，系统识别该错误类型，引导用户联系携程客服清空数据后重新同步。

**携程接口错误码**

| 错误码 | 描述 |
|--------|------|
| 0 | 请求成功 |
| 990 | 当前接口调用方式存在安全隐患，请改用https方式调用 |
| 1000 | 接口权限受限 |
| 302 | 身份验证失败 |
| 305 | 身份验证次数过期，请重新生成Ticket |
| 15601001 | 未启用组织结构，无法同步数据 |
| 15601002 | CustID为空或者为0 |
| 15601003 | ParentCustID为空或者与CustID相同 |
| 15601004 | ParentCustID无效 |
| 15601005 | CustID重复，已做去重处理 |
| 15601006 | CustID重复，且组织内容不相等同 |
| 15601007 | 没有有效的根节点 |
| 15601008 | 没有有效的组织信息 |
| 15601009 | 获取携程组织信息失败 |
| 15601010 | custName或父机构custName为空 |
| 15601011 | 数据更新失败 |
| 15601012 | 机构信息错误，请检查是否有循环依赖等 |
| 15601013 | 组织部门列表为空 |
| 15601014 | 已存在部门删除失败 |
| 15601015 | 部门删除 |
| 15601016 | 该部门未找到有效根部门 |
| 15601999 | 请参考异常信息 |

**错误处理对照表**

| 错误说明 | 处理方案 | 用户提示 |
|----------|----------|----------|
| 携程已有组织架构数据 | 弹出ID冲突弹窗，引导清空携程数据 | 同步失败：携程已有组织架构 |
| 携程组织接口同步失败 | 弹出tips提示 | xx原因同步失败 |

**ID冲突场景确认弹窗**：

```
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.3 同步重试机制

**功能概述**：用户按指引联系携程客服清空携程数据后，可打开"同步有成报销完整部门人员信息"开关再次触发同步。

**重新同步流程**：

```
用户再次打开"自动同步"开关或勾选"员工"点击"员工同步"
        ↓
  调用携程人事同步接口
        ↓
┌─────────────────┬─────────────────┐
│    同步成功      │    同步失败      │
└────────┬────────┴────────┬────────┘
         ↓                  ↓
  弹出tips"同步成功"  继续显示错误提示
  锁定"部门员工"同步   支持再次重试
```

#### 3.2.4 特殊说明

> 🔑 蓝色区域：xd测试（携程根节点组织，不允许删除）、默认部门（携程默认部门，存放没有部门id的员工，不允许删除）
> 红色区域：有成报销的根节点组织（推送到携程侧——本次需求不推送有成组织，直接使用携程根节点组织）、董事会/测试等（有成报销的部门，推送到携程侧）

**新功能推送逻辑重点**：
- 有成报销首次同步组织部门和员工信息至携程侧时，只要携程侧组织下返回了除默认部门以外的其他部门信息，就视为ID冲突，提示客户联系携程清除组织架构信息
- 有成报销顶级层级不同步至携程侧，使用携程侧顶级层级，原先挂在有成报销顶级层级下的员工，同步至携程侧后挂在携程侧默认部门下

### 3.3 逻辑互斥与边界处理

| 边界场景 | 触发条件 | 处理方案 |
|----------|----------|----------|
| 首次同步超时 | 接口60秒未返回 | 显示tips"同步超时"，支持再次打开"同步"开关，后台继续轮询 |
| 同步过程中有成数据变更 | 首次同步进行中有成组织架构变更 | 以同步发起时刻数据为准，完成后增量同步变更 |
| 清空不彻底再次同步失败 | 携程数据清空不彻底，ID仍冲突 | 继续提示ID冲突，建议再次联系携程客服。开关恢复关闭 |
| 网络异常 | 调用接口时网络中断 | 提示网络异常，保存同步任务，网络恢复后自动重试 |
| 携程接口变更错误码 | 携程返回未定义的错误码 | 归类为"未知错误"，提示联系客服，记录完整错误日志供研发排查 |

**与其他功能的兼容性**

| 相关功能 | 交互关系 | 处理方案 |
|----------|----------|----------|
| 钉钉组织架构同步 | 上游数据源 | 钉钉→有成→携程单向流转，钉钉变更触发有成变更，有成变更触发携程同步 |
| 有成组织架构调整 | 触发源 | 人事同步开启且成功后，有成组织架构部门员工变更通过"定时任务"或"员工同步"更新至携程商旅 |
| 应用中心其他模块 | 依赖组织架构 | 携程商旅其他功能（商旅预订等）依赖同步成功的组织架构数据和员工数据 |

## 附录

### A. 典型使用场景

**场景一：先开通携程，后开通有成（ID冲突解决）**

配置步骤：
1. 客户已使用携程商旅组织，有部门ID bm01
2. 开通有成报销，创建组织架构，生成组织ID 002和部门id bm001
3. 进入应用中心-携程商旅-员工管理-同步配置，勾选"同步有成报销完整部门人员信息"，启用"自动同步"或"员工同步"
4. 系统调用人事同步接口，返回ID冲突错误
5. 页面显示冲突提示：携程部门ID bm01 vs 部门id bm001
6. 按指引联系携程客服，申请清空组织架构数据
7. 客服清空完成后，再次点击"自动同步"或"员工同步"开关按钮
8. 同步成功，状态更新，携程更新组织架构内人员数据

结果：ID冲突解决，双方组织架构一致。

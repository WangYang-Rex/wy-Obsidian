## 1\. 需求背景

### 1.1 需求描述

有成报销现有金额组件在外币业务场景下存在精度误差问题：用户录入金额后，经系统多次汇率换算，在详情页、列表页、关联单据等环节出现金额不一致（如录入5000显示5000.03、关联单据可用金额变为4999.97）。需在不影响现有SaaS存量用户的前提下，通过分阶段迭代解决外币金额展示精度和尾差问题。

### 1.2 需求背景

**需求方**：企业财务管理员、出纳人员、审批人

**现状问题**：
- 当前系统DB统一存本位币，OUT展示时走"本位币÷汇率→原币"反算，产生精度误差
- 费用明细合计、分摊合计、关联单据扣减等计算链路中，多次换算累积尾差
- 涉及外币的单据在详情页、列表页、审批流、消息推送等21个OUT场景中，部分场景未同时展示原币和本位币

**解决思路**：
- 展示层：直接取DB原始值，不再反算；涉及外币的场景统一双币种展示
- 计算层：在"合计=明细之和"的汇总环节，尾差自动补足到最后一行
- 隔离策略：本位币≠报销币种时触发新逻辑，同币种单据完全不受影响

**业务流程**
```
阶段一/二（并行）：
单据录入/编辑/详情页双币种展示（展示层不再反算）
+ 费用计算尾差自动补足（计算层）
    ↓
阶段三：
列表页/审批流/任务列表双币种展示
    ↓
阶段四：
消息推送/报表/剩余OUT场景覆盖
    ↓
阶段五：
预算报表双币种改造（预算报表展示层双币种，预算控制逻辑不动）

```

## 2\. 功能现状

### 2.1 币种金额汇率转换现状

**位置**：全系统所有涉及金额展示和计算的模块

**现有能力**：
- IN方向（3个枚举）：单据提交、费用提交、费用标准校验时，原币×汇率→本位币存入DB
- OUT方向（21个枚举）：DB读取时，本位币÷汇率→原币反算展示
- 汇率随单据持久化（exchangeRate字段存入DB）
- 已有originPayAmount/originSubmitAmount等原币字段

**现有缺失**：
- OUT展示时部分场景走反算逻辑，未直接取originXxx原始值，导致精度误差
- 涉及外币的场景未全量覆盖双币种展示
- 费用合计/分摊/关联扣减等计算链路未做尾差校验和补足

### 2.2 多公司组织现状
- 同一租户下多个公司主体，部分公司有外币业务，部分公司无
- 公司主体维度有币种配置（本位币、报销币种）
- 无外币业务的公司：本位币=报销币种，不涉及汇率换算

## 3\. 迭代阶段规划

### 3.1 迭代总览

| 阶段 | 名称 | 目标 | 范围 | 优先级 | 预估工期 |
|------|------|------|------|---------|------------|
| Phase 1 | 单据详情/编辑页/打印双币种展示 | 解决最直观的展示精度问题 | 全部单据类型的详情页、编辑页、添加/编辑明细弹窗、单据打印模板 | P0 | 2周 |
| Phase 2 | 费用计算尾差补足 | 解决金额精度误差的根因 | 费用明细合计、费用分摊合计、关联单据扣减 | P0 | 2周 |
| Phase 3 | 列表页/审批流/任务列表/数据导出 | 高频查看场景统一双币种 | 单据管理列表、我的单据、审批页面、任务列表、Excel数据导出 | P1 | 2周 |
| Phase 4 | 消息推送/企业报表/项目报表/流程效率报表/自定义报表/剩余OUT场景 | 覆盖全部21个OUT枚举及普通报表 | 消息推送、首页报表、企业报表、项目报表、流程效率报表、自定义报表、借款、发票等 | P1 | 2周 |
| Phase 5 | 预算报表双币种改造 | 预算报表涉及预算控制逻辑，单独收尾 | 预算报表页面及预算报表Excel导出 | P2 | 2周 |

### 3.2 阶段间依赖关系
```
Phase 1（展示层-单据页）
    ↓ 依赖：Phase 1展示逻辑验证通过后，列表页可复用
Phase 3（展示层-列表/审批/任务）
    ↓ 依赖：全部展示层改造完成后，消息/报表统一收口
Phase 4（展示层-消息/普通报表/剩余OUT场景）

Phase 5（预算报表）
    ↓ 依赖：全部展示层和计算层改造完成后，预算控制逻辑才具备双币种基础

```
> Phase 1 和 Phase 2 可并行开发，互不依赖。   

### 3.3 前端展示方案设计（各场景统一规范）

涉及外币业务时，不同功能区的展示方式需根据**空间限制**和**业务场景**差异化设计，不能一刀切。

| 场景 | 空间特点 | 展示方案 | 格式示例 |
|------|------------|------------|------------|
| **单据详情/编辑页-主金额** | 空间充裕 | 双行上下排列，原币在上，本位币在下，标签区分 | `原币：USD 735.30``本位币：CNY 5,000.00` |
| **单据详情-费用明细表格** | 列宽有限 | 金额单元格内双行紧凑排列，字号统一 | `CNY 5,000.00``USD 735.30` |
| **列表页（单据/任务）** | 列宽极有限 | **默认显示本位币**，hover tooltip 展示原币；或紧凑括号格式 | 默认：`CNY 5,000.00` hover：`原币：USD 735.30` |
| **审批页面** | 空间充裕 | 同详情页，双行展示；审批金额以原币为准标注 | `USD 735.30``CNY 5,000.00``（审批金额以原币为准）` |
| **消息推送** | 空间极有限 | 紧凑单行格式，原币在前，本位币括号内 | `USD 735.30 (CNY 5,000.00)` |
| **打印模板（PDF/纸质）** | 横向空间充裕 | 单行横向排列，斜杠分隔，节省纵向空间 | `USD 735.30 / CNY 5,000.00` |
| **数据导出Excel** | 列宽充裕 | **拆成两列**：原币金额列 \+ 本位币金额列，便于后续Excel计算 | `原币金额：USD 735.30``本位币金额：CNY 5,000.00` |
| **普通报表（企业/项目/流程效率）** | 列宽有限 | 本位币金额列保持不动，**有条件增加"原币明细"列**：单种原币直接显示，多种原币分号分隔，无外币时隐藏 | `USD 2,205.90` 或 `USD 2,205.90; EUR 1,000.00` |
| **预算报表** | 同普通报表 | 同普通报表逻辑：本位币列 \+ 原币明细列 | 同普通报表 |
| **自定义报表** | 用户可控 | 设计器中金额字段增加「显示原币明细」开关，默认关闭，用户按需开启 | 关闭：单币种 / 开启：本位币\+原币明细 |
| **金额组件（binding）** | 表单字段区域 | 同单据详情页主金额：双行上下排列 | `USD 735.30``CNY 5,000.00` |
| **关联单据列表** | 列宽有限 | 同列表页逻辑：默认本位币，hover展示原币 | 默认：`CNY 5,000.00` hover：`原币：USD 735.30` |
| **首页报表卡片** | 卡片空间有限 | 紧凑格式：本位币为主，小字标注原币 | `CNY 5,000.00``（USD 735.30）` |
| **借款记录/还款记录** | 列表页 | 同列表页逻辑 | 默认：`CNY 5,000.00` hover：`原币：USD 735.30` |
| **历史单据** | 同详情页 | 同详情页逻辑 | 同详情页 |
| **凭证科目映射选择页** | 选择列表 | 同列表页逻辑 | 默认：`CNY 5,000.00` hover：`原币：USD 735.30` |

**展示方案核心原则**
1. **空间充裕场景**（详情页、编辑页、审批页、打印）：双币种完整展示，不依赖交互
2. **空间受限场景**（列表页、任务列表、消息）：默认显示本位币，通过hover/括号/小字补充原币信息
3. **数据处理场景**（Excel导出）：拆成独立列，便于用户后续计算
4. **聚合统计场景**（报表）：原币按币种分组展示，避免不同币种直接相加的数学错误
5. **用户可控场景**（自定义报表、打印模板）：提供开关/配置，默认关闭保证兼容

## 4\. 各阶段详细说明

### 4.1 Phase 1：单据详情/编辑页/打印双币种展示改造（P0）

**阶段目标**

解决用户截图中最直观的展示问题：付款单主金额显示5000.03、关联单据可用金额显示4999.97。确保单据详情页、编辑页、单据打印中所有金额字段直接取DB原始值，不再做反算。

**覆盖范围**

| 单据类型 | 页面 |
|------------|------|
| 报销单 | 详情页、编辑页、添加费用明细弹窗、打印输出 |
| 借款单 | 详情页、编辑页、添加费用明细弹窗、打印输出 |
| 付款单 | 详情页、编辑页、添加费用明细弹窗、收款信息、打印输出 |
| 收款单 | 详情页、编辑页、添加费用明细弹窗、打印输出 |
| 合同 | 详情页、编辑页、合同支付计划、打印输出 |

**核心改造点**

| 改造项 | 现有逻辑 | 目标逻辑 |
|---------|------------|------------|
| 金额展示 | 本位币÷汇率反算展示 | 直接取DB原始值（payAmount/submitAmount/originXxx），不做任何换算 |
| 双币种展示 | 部分场景只有单币种 | 本位币≠报销币种时，统一显示「原币币种\+原币金额 / 本位币币种\+本位币金额」 |
| 金额组件binding | OUT时val可能被替换为反算值 | OUT时val直接取originAmount原始值 |
| 费用明细金额 | 单币种展示 | 双币种上下排列展示，原币在上，本位币在下 |
| 单据打印模板 | 打印时金额字段单币种展示 | 打印模板金额字段支持双币种展示，格式：原币 / 本位币 |
| 打印输出渲染 | 金额字段按单币种渲染 | 涉及外币单据打印时，金额字段同时渲染原币\+本位币 |

**系统提示文案**
- 首次外币单据提示：「本单据涉及外币，金额将同时展示原币和本位币。」
- 金额字段hover：「原币：录入时的原始金额；本位币：按当时汇率换算后的金额。」

**验收标准**

| 序号 | 验收项 | 验收标准 |
|------|---------|------------|
| 1 | 付款单详情页金额 | 录入CNY 5000，详情页显示CNY 5,000.00，不出现5000.03 |
| 2 | 关联单据可用金额 | 关联扣减后，可用金额显示正确，不出现4999.97 |
| 3 | 双币种展示 | 本位币≠报销币种的单据，详情页所有金额字段同时显示原币\+本位币 |
| 4 | 同币种单据 | 本位币=报销币种的单据，保持现有单币种展示，无任何变化 |
| 5 | 金额组件 | 自定义金额组件（binding）展示时直接取originAmount，不再反算 |
| 6 | 单据打印 | 涉及外币的单据打印输出，金额字段同时显示原币和本位币，如 USD 735.30 / CNY 5,000.00 |
| 7 | 打印模板设计器 | 打印模板中金额字段支持配置「单币种展示」或「双币种展示」，默认单币种（兼容存量） |

### 4.2 Phase 2：费用计算尾差补足（P0）

**阶段目标**

解决金额精度误差的根因。在费用明细合计、费用分摊合计、关联单据扣减等计算链路的汇总环节，如果因汇率精度产生尾差，自动将尾差补足到最后一行费用的本位币金额。

**覆盖范围**

| 计算场景 | 说明 |
|------------|------|
| 费用明细合计 | 费用明细本位币之和 vs 单据总金额 |
| 费用分摊合计 | 分摊明细本位币之和 vs 费用金额 |
| 关联单据扣减 | 扣减后可用金额 vs 原金额-扣减金额 |
| 借款核销 | 核销金额合计 vs 借款金额 |

**核心改造点**

| 改造项 | 现有逻辑 | 目标逻辑 |
|---------|------------|------------|
| 费用合计 | 各费用本位币直接相加 | 相加后校验是否等于单据总金额，不等则尾差补到最后一行 |
| 费用分摊 | 分摊金额直接换算后相加 | 相加后校验是否等于费用金额，不等则尾差补到最后一条分摊 |
| 关联扣减 | 直接扣减 | 扣减后校验可用金额，不等则尾差补到最后一条扣减 |
| 尾差阈值 | 无 | 绝对值≤0.1时自动补足，超过抛异常 |

**尾差补足规则**
- 只调整**本位币**，原币金额不动
- 补足到**最后一行**（或金额最大的一行）
- 补足后：明细之和 = 合计值
- 补足记录写入操作日志，便于审计

**异常提醒文案**
- 尾差超过阈值：「汇率换算尾差超过允许范围（\>0.1），请检查汇率设置或联系管理员。」

**验收标准**

| 序号 | 验收项 | 验收标准 |
|------|---------|------------|
| 1 | 费用明细合计 | 3条费用各1000，合计应为3000，因精度算出2999.97时自动补足到最后一行 |
| 2 | 费用分摊合计 | 分摊3行各333.33，合计应为1000，算出999.99时自动补足到最后一条分摊 |
| 3 | 关联单据扣减 | 可用金额5000，扣减5000后应为0，因精度变成-0.03时自动补足 |
| 4 | 尾差阈值 | 尾差0.08时自动补足，尾差0.15时抛异常 |
| 5 | 同币种单据 | 本位币=报销币种时，不触发任何尾差补足逻辑 |

| 流程效率报表 | 审批效率相关金额字段 | — |
| 自定义报表 | 用户自定义配置的金额字段 | — |
| 借款管理 | 借款记录列表、借款详情、还款记录 | LOAN\_RECORD\_LIST\_OUT / LOAN\_RECORD\_RESPONSE\_OUT |
| 发票管理 | 待收发票列表 | invoiceToBeReceived\_OUT |
| 合同管理 | 合同列表、合同应付款、合同选择 | Contract\_Order\_Page\_OUT 等 |
| 凭证科目映射 | 凭证生成前选择单据 | REIMBURSE\_ORDER\_MAP\_OUT 等 |
| 历史单据 | 查看单据历史版本 | ORDER\_HISTORY\_OUT |

**核心改造点**

| 改造项 | 目标逻辑 |
|---------|------------|
| 消息推送 | 消息体中金额字段同时携带原币和本位币，格式：USD 735.30 (CNY 5,000.00) |
| 普通报表（企业/项目/流程效率/自定义） | 报表金额列保持本位币展示，有条件增加"原币明细"列：单种原币直接显示，多种原币按币种分组用分号分隔，无外币时列隐藏 |
| 报表Excel导出 | 导出时在本位币金额列旁增加"原币明细"列，格式与页面一致 |
| 自定义报表设计器 | 金额字段增加"显示原币明细"开关，默认关闭 |
| 剩余OUT场景 | 全部走"直接取DB原始值，不再反算"逻辑 |

**验收标准**

| 序号 | 验收项 | 验收标准 |
|------|---------|------------|
| 1 | 消息推送 | 钉钉/飞书/企微消息中涉及外币单据的金额，同时显示原币和本位币 |
| 2 | 首页报表 | 首页金额统计卡片涉及外币时显示双币种 |
| 3 | 企业/项目/流程效率报表 | 涉及外币的报表行，金额旁增加原币明细列，格式正确 |
| 4 | 自定义报表 | 开启"显示原币明细"后，报表正确展示原币明细；关闭后保持原样 |
| 5 | 借款记录 | 借款记录列表和详情页双币种展示 |
| 6 | 全量OUT覆盖 | 全部21个OUT枚举场景中，涉及外币的金额展示正确无误 |
| 7 | 打印模板兼容性 | 存量打印模板未开启双币种配置时，打印输出保持原有单币种格式 |

### 4.5 Phase 5：预算报表双币种改造（P2）

**阶段目标**

预算报表涉及预算金额、预算执行金额、预算余额等核心字段，且预算控制逻辑（预算是否超支）与金额展示紧密耦合。因此预算报表单独放在最后一步，待全部展示层和计算层改造完成后，再基于稳定的双币种金额数据进行预算报表的双币种展示改造。
> **核心原则**：预算控制逻辑（是否超支、是否允许提交）**仍按本位币执行**，预算报表仅做展示层双币种改造，不动预算控制核心逻辑。   

**覆盖范围**

| 场景 | 说明 |
|------|------|
| 预算报表页面 | 预算执行报表、预算对比报表、预算余额报表等 |
| 预算报表Excel导出 | 预算报表导出Excel |

**核心改造点**

| 改造项 | 现有逻辑 | 目标逻辑 |
|---------|------------|------------|
| 预算报表展示 | 预算金额、执行金额、余额仅显示本位币 | 金额列保持本位币展示，有条件增加"原币明细"列（同Phase 4报表逻辑） |
| 预算控制逻辑 | 按本位币判断预算是否超支 | **保持现状**，预算控制仍按本位币执行，不受改造影响 |
| 预算报表导出 | 仅导出本位币 | 导出Excel时增加"原币明细"列 |

**关键约束**
- 预算控制（提交单据时判断是否超预算）**必须按本位币执行**，不能按原币控制
- 原因：不同单据的原币可能不同（USD、EUR等），预算额度是按本位币（如CNY）制定的，按原币控制无统一标准
- 预算报表仅做展示，让用户看到预算执行的原币构成，但控制门槛仍是本位币

**验收标准**

| 序号 | 验收项 | 验收标准 |
|------|---------|------------|
| 1 | 预算报表双币种展示 | 预算报表中涉及外币的金额行，正确显示原币明细列 |
| 2 | 预算控制不变 | 预算控制逻辑（超支判断、提交拦截）完全保持现有本位币逻辑，不受改造影响 |
| 3 | 预算报表导出 | 预算报表Excel导出时，原币明细列格式正确 |
| 4 | 单币种预算 | 无外币业务的预算报表，保持现有展示，无任何变化 |

## 5\. 风险与回滚策略

### 5.1 阶段风险

| 阶段 | 风险点 | 缓解措施 |
|------|---------|------------|
| Phase 1 | 展示层改造影响同币种单据 | 严格做币种判断，本位币=报销币种时走原有逻辑分支 |
| Phase 2 | 尾差补足影响费用标准校验 | 尾差补足发生在费用标准校验之后，或在校验逻辑中排除尾差调整行 |
| Phase 3 | 列表页性能下降（双币种查询） | 列表页查询时直接取DB原始值，不做实时换算，性能影响可控 |
| Phase 4 | 消息模板改造影响存量企业 | 消息模板增加字段但不删除旧字段，存量企业消息格式兼容 |
| Phase 5 | 预算报表涉及预算控制逻辑改造 | 预算控制需确认按本位币还是原币执行，改造前需对齐规则 |

### 5.2 回滚策略

| 场景 | 回滚方案 |
|------|------------|
| Phase 1/2 上线后异常 | 展示层和计算层改造均有币种判断隔离，同币种单据完全不受影响；外币单据可临时关闭双币种展示开关（如需要） |
| Phase 5 预算控制规则冲突 | 预算报表改造前，先确认预算控制按本位币执行，预算报表仅做展示层双币种，不动控制逻辑 |

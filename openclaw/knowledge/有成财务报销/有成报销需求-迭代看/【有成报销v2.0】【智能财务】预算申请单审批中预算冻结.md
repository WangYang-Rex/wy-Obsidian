## 〇、现有逻辑说明（供参考）

### 0.1 预算预占用机制现状
- **报销单/付款单/借款单**：提交送审时预占用预算（order\_budget\_info.status=1），审批通过后转为已使用（status=2），驳回/撤回后释放（status=3）
- **超额校验**：剩余可用 = 预算总额 - 已使用金额 - 预占用金额（基于周期粒度）
- **关键限制**：现有系统只在单据提交时做超额校验（excess()），审批通过/拒绝时仅做状态变更，**审批过程中不做超额校验**
- **预算申请单**：目前无冻结机制，提交后直接进入审批流，变更额度在审批期间可被其他单据预占用
> 本次需求需兼容现有预占用/释放逻辑，新增冻结金额参与校验。   

### 0.2 预算申请现有能力

| 能力 | 说明 | 是否冻结 |
|------|------|------------|
| **追加模式** | 新旧金额差额叠加到现有预算 | ❌ 不冻结 |
| **修改-金额上调** | 直接替换为新金额 | ❌ 不冻结 |
| **修改-金额下调/调减** | 直接替换为新金额，预算减少 | ✅ **冻结** |
| **新增维度** | 新增一个预算规则维度 | ❌ 不冻结 |
> **结论**：当前系统不支持跨维度调整，冻结仅针对"同维度金额下调（调减）"。

## 一、快速概览

### 1.1 这个需求要做什么？

**一句话**：预算申请单调减提交后审批期间，冻结目标周期额度，防止其他单据超额预占用。

**具体效果**：
- 同维度调减提交后，目标周期额度被冻结，其他单据无法继续消耗
- 审批通过后冻结金额生效并释放，驳回/撤回/超时后释放冻结
- 调减生效后自动兜底校验，标记超额预占用单据
- 预算看板可查看「冻结中」额度
- 追加/金额上调/新增维度不冻结
> **追加/上调不冻结的理由**：增加预算不影响当前可用余额。若冻结当前余额，反而让预算更紧张。   

### 1.2 谁需要这个功能？

| 角色 | 痛点 | 解决方案 |
|------|------|------------|
| 企业财务管理员 | 预算调减审批期间，报销单继续提交导致调减后实际超支 | 调减提交即冻结目标周期额度 |
| 审批人 | 审批通过后发现预算已超，需手动追回转审 | 调减生效前即保护剩余额度 |

### 1.3 核心价值
- ✅ 调减审批期间预算不被"透支"，预算控制实时准确
- ✅ 调减生效后自动识别超额单据，减少财务人工排查
- ✅ 冻结粒度落到周期维度，不影响其他周期单据

## 二、功能需求

### 2.1 冻结触发与计算

**冻结触发条件**：仅"修改模式\+金额下调"触发冻结

![image.png](https://alidocs2.oss-cn-zhangjiakou.aliyuncs.com/res/4j6OJ5jkbjwBKq3p/img/2b28491a-62a2-4098-8763-3019a393513d.png?Expires=1778231607&OSSAccessKeyId=LTAI5tKTjg4Kq1HCdBJ8qpSp&Signature=Lw1EKycLa7VlYDhbxYioWtnvMJM%3D "")

| 申请单类型 | 冻结行为 | 生效后预算变化 |
|---------------|------------|---------------------|
| 追加 | 不冻结 | 预算总额叠加差额 |
| 修改-金额上调 | 不冻结 | 预算总额替换为新金额 |
| 修改-金额下调/调减 | 冻结调减差额 | 预算总额替换为新金额，冻结释放 |
| 新增维度 | 不冻结 | 直接新增维度 |

**冻结金额计算**：冻结金额 = 原金额 - 新金额（调减差额）
> **冻结粒度**：按预算维度\+年度\+周期的最小粒度冻结，确保5月冻结不影响6月。   

### 2.2 冻结生命周期

| 已生效 | 调减审批通过 | 预算总额替换为新金额，冻结金额释放，冻结记录标记已生效 |
| 已释放 | 驳回/撤回/超时 | 冻结金额释放，冻结记录标记已释放 |

### 2.3 冻结期间的预算校验

**单据提交时的前置校验**：
- 报销单/付款单/借款单提交时，校验剩余可用预算
- 校验公式：剩余可用 = 预算总额 - 已使用金额 - 预占用金额 - 冻结中金额
- 不满足则拦截提交，超额提示按系统现有逻辑提示不变

**审批条件判断**（与超额校验是不同逻辑）：

![image.png](https://alidocs2.oss-cn-zhangjiakou.aliyuncs.com/res/4j6OJ5jkbjwBKq3p/img/91763bc7-01a0-49ad-b152-4d82046137bf.png?Expires=1778231607&OSSAccessKeyId=LTAI5tKTjg4Kq1HCdBJ8qpSp&Signature=THAfh%2BkZhNwapYTLHlO6zOfSx8g%3D "")
- 预算条件审批分支（如"预算使用率超过80%走特殊分支"）需纳入冻结金额
- 使用率 = (已使用金额 \+ 预占用金额 \+ 冻结中金额) / 预算总额
- **注意**：超额校验在单据提交时执行（判定能否提交），审批条件判断在审批流节点流转时执行（判定走哪个分支）

### 2.4 调减生效时的兜底校验

**触发时机**：调减审批通过后，预算总额已替换为新金额，冻结已释放

**校验逻辑**（直接累计判定，不复用提交时的 excess() 方法）：
1. 扫描该周期下所有审批中的单据（预占用状态）
2. 按提交时间从早到晚排序，逐张累加预占用金额
3. 新预算剩余可用 = 新预算总额 - 已使用金额
4. 逐张判定：累加金额 \> 新预算剩余可用 ？ 
    - 当前单据累加后首次超过 → 标记当前及之后的所有单据为"预算超额"
    - 累加始终未超过 → 无操作
> **为什么按提交顺序？** 遵循"先到先得"原则。单据提交时已通过前置校验（当时预算充足），调减后预算下降导致后续单据累计超额。后提交的单据在提交时即使通过了校验（冻结金额已扣减），调减生效后仍可能因预算下降而实际超额。按顺序判定可精准定位真正导致超支的单据，避免先提交的单据被误标。   

**超额处理**：按企业配置的超额策略执行
> **关键差异说明**：现有强控制逻辑是"单据提交时超额→禁止提交"。本次场景中，单据提交时预算充足，是调减生效后才超额的（单据已在审批中），因此需要补充"审批中单据超额"的处理逻辑。   
- **强控制（禁止）**： 
    - 超额单据被系统标记"预算超额"状态
    - 当前审批人审批时，点击"同意"系统弹出提示弹窗（无标题，仅纯提示）：

**弹窗内容**：该单据所在预算周期「\{维度名称\}-\{周期\}」已发生调减，当前单据预占用金额超出新预算，禁止审批通过请驳回。
    - 当前审批人仅可选择"驳回"，不可同意；使用单据详情页已有的"驳回"功能驳回单据
    - 系统同步通知申请人："您提交的单据因预算调减已超额，请联系审批人或撤回重提"
    - **注意**：此拦截能力是本次需求新增，现有系统无"审批中超额拦截"机制
    - 申请人可在单据列表和单据详情查看"预算超额"标记，按照系统原超额显示逻辑不变
- **弱控制（提示）**：超额单据正常流转，审批页面显示"预算超额"警告标识，审批人可继续同意
- **不提示**：仅标记，单据正常流转，无感知

## 四、附录

### 4.1 典型使用场景

**场景1：调减预算导致周期超额**
1. 维度X-2026年5月预算总额10000，已用8000，冻结0
2. 提交调减3000（原10000→新7000）→ 前置校验：剩余2000 \< 3000 → 拦截
3. 改为调减1500（原10000→新8500）→ 校验通过
4. 审批通过 → 预算总额=8500，冻结金额=0
5. 兜底校验：累计预占用8000 \<= 8500 → 无超额

**场景2：追加后调减**
1. 维度X-2026年5月预算总额10000，已用8000
2. 提交追加2000 → 不冻结
3. 追加审批通过 → 预算总额=12000
4. 同月又提交调减3000（原12000→新9000）→ 冻结3000
5. 调减审批通过 → 预算总额=9000，冻结金额=0
6. 兜底校验：累计预占用8000 \<= 9000 → 无超额

**场景3：先预占用后调减导致超额（场景A）**
1. 维度X-2026年5月预算总额10000，已用6000，预占用0，冻结0
2. 报销单A提交（预占用2000）→ 预占用金额=2000，剩余可用=2000
3. 预算申请调减单提交（10000→7000）→ 冻结金额=3000
4. 调减审批通过 → 预算总额=7000，冻结金额=0
5. 兜底校验：新剩余可用=7000-6000=1000。A占2000，按顺序累加：2000\>1000 → A超额
6. 报销单A被标记"预算超额"，按超额策略处理
> **变体 — 多张单据时按顺序判定**：   

**场景4：调减期间新单据正常流转（场景B）**
1. 维度X-2026年5月预算总额10000，已用6000
2. 预算申请调减单提交（10000→8000）→ 冻结金额=2000
3. 报销单B提交（申请预占用1500）： 
    - 前置校验：剩余=2000 \>= 1500 → 通过
4. 调减审批通过 → 预算总额=8000，冻结金额=0
5. 兜底校验：累计预占用1500 \<= (8000-6000)=2000 → 不超额
6. 报销单B正常审批流转

**场景5：调减期间新单据临界拦截（场景B变体）**
1. 同上，调减单冻结2000
2. 报销单C提交（申请预占用2500）： 
    - 前置校验：剩余=2000 \< 2500 → 拦截
    - 系统提示"当前周期剩余预算2000元，本次申请预占用2500元，超出500元"
3. 用户改为申请预占用1500 → 校验通过

### 4.2 方案决策记录

| 决策项 | 结论 | 决策依据 |
|---------|------|------------|
| 追加是否冻结 | 不冻结 | 追加是叠加现有预算，冻结当前余额会让预算更紧张 |
| 金额上调是否冻结 | 不冻结 | 上调是直接替换为新金额，同样增加预算 |
| 调减是否冻结 | 冻结 | 下调减少预算，需保护剩余额度不被继续消耗 |
| 跨维度调整 | 不支持 | 当前系统不支持跨维度调整，仅支持同维度金额调整 |
| 冻结粒度 | 周期粒度 | 与现有预占用/校验体系保持一致 |
| 超额判定逻辑 | 累计预占用判定 | 与现有汇总逻辑一致，避免逐张扣除自身 |
| 动态预算 | 本次排除 | 动态预算由公式异步计算，冻结机制需联动公式引擎，复杂度超出本期范围 |
| 超时释放 | 30天自动释放 | 防止长期审批导致预算长期不可用 |

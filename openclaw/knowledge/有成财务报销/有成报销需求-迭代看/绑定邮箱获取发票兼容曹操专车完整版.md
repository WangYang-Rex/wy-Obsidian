## 〇、现有逻辑说明（供参考）

### 0.1 邮箱拉取发票现有流程

有成报销已支持绑定邮箱自动拉取发票生成消费，三阶段异步流水线：
- **阶段一（邮件拉取）**：
    - 附件型邮件：提取附件 → 按 **PDF \> OFD \> XML** 优先级仅保存一种格式 → 上传OSS → 保存 `email_attachment`（邮件附件表）
    - 正文链接型邮件：仅支持**诺诺网**、**阿里发票平台**，解析HTML获取下载链接 → 按 **PDF \> OFD \> XML** 优先级仅保存一种格式 → 下载 → 上传OSS
- **阶段二（OCR识别）**：`MailAttachmentToInvoiceTask`（附件转发票定时任务）遍历 `email_attachment`（邮件附件表），调用 `financeInvoiceList.rjson`（OCR发票识别接口）识别发票
- **阶段三（消费生成）**：`MailInvoiceToFeeTask`（发票转消费定时任务）按用户配置的单条/合并模式生成消费记录

### 0.2 正文链接型邮件现有支持

| 平台 | 识别方式 | 处理逻辑 |
|------|------------|------------|
| 诺诺网 | 发件人含"诺诺网" | 解析HTML获取发票下载链接 |
| 阿里发票平台 | 发件人含"阿里发票平台" | 解析HTML获取发票下载链接 |
| 其他平台 | — | ❌ 不处理，发票无法提取 |

### 0.3 行程单现有处理逻辑

合并模式（`emailInvoiceMergeToFee`（邮件发票合并转消费配置）== 1）下：
- 附件文件名含"行程" 或 "高德打车\+行程单" → `shouldIgnoreAttachmentForMerge()`（合并模式附件忽略判断）跳过OCR
- 被跳过的附件 `invoiceId`（发票ID）= null，在 `invoiceToFeeMerge()`（发票转消费合并方法）中被收集到 `attachmentList`（附件列表）
- 最终作为**非发票附件**挂载到合并消费记录的附件组件（`ComponentTypeEnum.ATTACHMENT`（组件类型枚举-附件））

## 一、快速概览

### 1.1 这个需求要做什么？

**一句话**：在现有绑定邮箱获取发票生成消费的功能基础上，新增对\*\*曹操专车（曹操出行）\*\*发票邮件的兼容支持。

**具体效果**：
- 用户绑定邮箱后，收到曹操出行电子发票邮件，系统自动识别、下载发票PDF、OCR识别、生成消费记录
- 同时下载邮件中的行程单，作为非发票附件挂载到消费记录
- 无需用户手动操作，与现有滴滴/高德体验一致

### 1.2 谁需要这个功能？

| 角色 | 痛点 | 解决方案 |
|------|------|------------|
| 企业员工 | 使用曹操专车后需手动下载发票并上传到报销系统 | 绑定邮箱后自动拉取，零手动操作 |
| 财务管理员 | 曹操专车发票无法被系统自动采集，员工常遗漏 | 系统自动识别曹操邮件，发票不遗漏 |

### 1.3 核心价值
- ✅ 曹操专车发票自动采集，减少员工手动操作
- ✅ 行程单同步下载，满足财务报销完整凭证要求
- ✅ 复用现有三阶段流水线，改动范围可控

## 二、功能需求

### 2.1 曹操专车邮件识别与解析

**位置**：阶段一（邮件拉取）— `EmailTaskProcessServiceImpl.doEmailPullAllTask()`（邮件任务处理服务-执行全部邮件拉取任务）

![image.png](https://alidocs2.oss-cn-zhangjiakou.aliyuncs.com/res/Yvenve510YvG1loy/img/1ec1b24e-700c-42fa-a3f4-708b14ca94f5.png?Expires=1778767559&OSSAccessKeyId=LTAI5tKTjg4Kq1HCdBJ8qpSp&Signature=1CsugnfuuV2Tt3Bc2gWu%2FbXtb8Y%3D "")

**新增识别规则**：

| 判断维度 | 规则 |
|------------|------|
| 发件人 | 包含 **"caocao"**（不区分大小写）或 **"曹操出行"** |
| 邮件标题 | 包含 **"发票"** 或 **"电子票"**（复用现有标题过滤） |
> 双重校验：发件人含"caocao"/"曹操出行" **且** 标题含"发票"/"电子票"，才判定为曹操出行发票邮件。   

**HTML正文解析**：

解析邮件正文HTML，提取以下两类链接：

| 链接类型 | 提取关键词 | 处理方式 |
|------------|---------------|------------|
| **发票PDF文件** | 正文含"发票PDF文件"或类似锚点 | 下载 → 上传OSS → `email&#95;attachment`（邮件附件表，`invoice&#95;status`（发票状态）=0）→ 走OCR |
| **所关联行程单** | 正文含"所关联行程单"或类似锚点 | 下载 → 上传OSS → `email&#95;attachment`（邮件附件表，`invoice&#95;status`（发票状态）=0）→ **跳过OCR** |
> **多格式优先级处理**：同一发票存在 PDF \+ OFD \+ XML 多种格式链接时，按 **PDF \> OFD \> XML** 优先级，**仅下载优先级最高的一种**，其余跳过。示例：曹操邮件同时提供 PDF、OFD、XML 三种发票链接 → 仅下载 PDF，OFD 和 XML 跳过。   

### 2.2 行程单文件名处理

**目的**：确保行程单能被现有 `shouldIgnoreAttachmentForMerge()` 规则识别，复用现有逻辑。

**规则**：
- 下载行程单文件后，保存到 `email_attachment` 时
- 若原始文件名**不含"行程"或"行程单"**，则在文件名前追加前缀 **"曹操出行行程单\_"**
- 若原始文件名已含"行程"或"行程单"\*\*，保持原文件名不变

**示例**：

| 原始文件名 | 处理后文件名 | 说明 |
|---------------|------------------|------|
| `2740482198513913.pdf` | `曹操出行行程单&#95;2740482198513913.pdf` | 追加前缀，确保被识别 |
| `曹操出行行程单.pdf` | `曹操出行行程单.pdf` | 已含关键字，不变 |

### 2.3 行程单OCR跳过逻辑

**位置**：阶段二（OCR识别）— `EmailAttachmentServiceImpl.handleeAttachmentTask()`（邮件附件服务-处理附件任务）

**规则**：
- 遍历 `email_attachment`（邮件附件表）时，若记录来源为曹操出行行程单（通过文件名含"曹操出行行程单"或"行程单"且关联的 `email_pull_record.from_user_name`（邮件拉取记录-发件人名称）含"caocao"判定）
- **直接跳过OCR**，不调用 `financeInvoiceList.rjson`（OCR发票识别接口）
- `invoice_status`（发票状态）不更新（保持初始值或设为特殊标记），`fee_status`（消费状态）设为 **WAIT(3)**
- 进入阶段三消费生成时，作为 `attachmentList`（附件列表）被挂载
> 简化方案：行程单文件名经2.2处理后已含"行程单"，可被现有 `shouldIgnoreAttachmentForMerge()` 自动识别并跳过OCR（合并模式下）。单条模式下同理处理。   

### 2.4 消费生成（复用现有逻辑）

![image.png](https://alidocs2.oss-cn-zhangjiakou.aliyuncs.com/res/Yvenve510YvG1loy/img/da2afee4-1d99-40cd-9b6f-45d965519ff7.png?Expires=1778767559&OSSAccessKeyId=LTAI5tKTjg4Kq1HCdBJ8qpSp&Signature=zt6oSzlLziMgSxHqYR1xubpOW0w%3D "")

**单条模式**（`emailInvoiceMergeToFee`（邮件发票合并转消费配置）== 0）：
- 发票PDF → 正常OCR → 生成一条消费记录
- 行程单 → 跳过OCR → 作为附件挂载到该条消费记录

**合并模式**（`emailInvoiceMergeToFee`（邮件发票合并转消费配置）== 1）：
- 以 `emailPullId`（邮件拉取ID）为维度，同一封曹操邮件的发票PDF \+ 行程单合并为**一条消费记录**
- 发票PDF → OCR成功 → 发票组件数据
- 行程单 → `invoiceId`（发票ID）= null → 收集到 `attachmentList`（附件列表）→ 附件组件数据

## 四、附录

### 4.1 曹操专车邮件特征总结

| 特征项 | 值 |
|---------|---|
| 发件人 | `notify@mail.caocaokeji.cn`（核心识别特征：`caocao`） |
| 主题 | `曹操出行电子发票` |
| 发票获取方式 | 正文HTML下载链接（非附件） |
| 发票格式 | PDF \+ XML \+ OFD（3种链接，按PDF\>OFD\>XML优先级仅保存一种） |
| 行程单 | 单独链接「所关联行程单」 |
| 链接有效期 | 行程单标注72小时（发票链接待确认） |

### 4.2 现有平台支持对比

| 平台 | 邮件类型 | 现有支持 | 识别特征 |
|------|------------|------------|------------|
| 滴滴 | 附件型（PDF） | ✅ 已支持 | 无专门识别，走通用附件流程 |
| 高德 | 附件型（PDF\+XML/OFD） | ✅ 已支持 | 文件名含"高德打车"，PDF/XML去重 |
| 诺诺网 | 正文链接型 | ✅ 已支持 | 发件人含"诺诺网" |
| 阿里发票平台 | 正文链接型 | ✅ 已支持 | 发件人含"阿里发票平台" |
| **曹操出行** | **正文链接型** | **❌ 本次新增** | **发件人含"caocao"** |

### 4.3 改动范围总结

| 阶段 | 改动点 | 工作量评估 |
|------|---------|---------------|
| 阶段一（邮件拉取） | 新增曹操发件人识别 \+ HTML解析（提取PDF链接\+行程单链接）\+ XML/OFD跳过 \+ 行程单文件名处理 | 中等 |
| 阶段二（OCR识别） | 复用现有`shouldIgnoreAttachmentForMerge()`（合并模式附件忽略判断，行程单文件名已处理） | 无改动或极小 |
| 阶段三（消费生成） | 复用现有单条/合并模式逻辑 | 无改动 |
